## OOP Nima — Paradigma Sifatida

### Nazariya

OOP — bu dasturlash paradigmasi bo'lib, dasturni **ob'ektlar** atrofida tashkil qiladi. Har bir ob'ekt o'z ichiga **ma'lumot** (properties/state) va shu ma'lumot ustida ishlaydigan **xatti-harakatlar** (methods/behavior) ni oladi.

OOP ning asosiy g'oyasi — real dunyodagi narsalarni kodda model qilish. Masalan, "Foydalanuvchi" ob'ekti `name`, `email` kabi ma'lumotlarga va `login()`, `logout()` kabi xatti-haratkatlarga ega.

OOP 4 ta asosiy ustun (pillar) ga qurilgan — ularning har biri alohida muammoni yechadi:

```
┌───────────────────────────────────────────────────────────────┐
│                    OOP — 4 USTUN                              │
├──────────────────┬──────────────────┬─────────────┬───────────┤
│   ENCAPSULATION  │   ABSTRACTION    │ INHERITANCE │POLYMORPHISM│
│                  │                  │             │           │
│ Ma'lumotni       │ Murakkablikni    │ Kodni qayta │ Bir xil   │
│ himoyalash,      │ yashirish,       │ ishlatish,  │ interface,│
│ ichki state ni   │ faqat kerakli    │ ota-bola    │ turli xil │
│ tashqi koddan    │ qismini          │ munosabat   │ xatti-    │
│ berkitish        │ ko'rsatish       │ qurish      │ harakat   │
└──────────────────┴──────────────────┴─────────────┴───────────┘
```

JavaScript — **multi-paradigm** til. U OOP ni ham, functional programming ni ham, procedural ni ham qo'llab-quvvatlaydi. Lekin JS dagi OOP **prototypal** — ya'ni klassik class-based tillardan (Java, C#) farqli o'laroq, meros **prototype chain** orqali ishlaydi.

```javascript
const userEmail = "ali@mail.com";

  userBalance += amount;
}

userBalance = -999; // hech kim to'xtatmaydi

    this.owner = owner;
    this.email = email;
    this.#balance = initialBalance;
  }

    return this.#balance;
  }

    if (amount <= 0) throw new RangeError("Musbat son kiriting");
    this.#balance += amount;

    if (amount > this.#balance) throw new Error("Mablag' yetarli emas");
    this.#balance -= amount;
  }
}

const account = new BankAccount("Ali", "ali@mail.com", 100_000);
account.deposit(50_000);
// account.#balance = -999; // ❌ SyntaxError — himoyalangan!
```

## OOP 4 Ustuni

### 1. Encapsulation

#### Nazariya

Encapsulation — ob'ektning ichki state'ini tashqi koddan **berkitish** va faqat aniq belgilangan method'lar orqali kirish imkonini berish. Bu ikkita asosiy g'oyani o'z ichiga oladi:

1. **Data va behavior ni birga saqlash** — ob'ekt ichida
2. **Ichki implementatsiyani yashirish** — tashqi kod faqat public API bilan ishlaydi

JS da encapsulation qilishning 4 ta usuli bor:

```
┌─────────────────────────────────────────────────────────┐
│            JS DA ENCAPSULATION USULLARI                 │
├───────────────────────┬─────────────────────────────────┤
│  1. Closures          │ function scope orqali yashirish │
│  2. Private fields #  │ til darajasidagi enforcement    │
│  3. WeakMap pattern   │ class dan tashqarida saqlash    │
│  4. Symbols           │ non-enumerable, qiyin kirish    │
└───────────────────────┴─────────────────────────────────┘
```

<details>
<summary><strong>Under the Hood</strong></summary>

V8 da private fields (`#`) **PrivateName** deb ataladigan maxsus internal symbol orqali ishlaydi. Har bir `#field` uchun class definition vaqtida noyob `PrivateName` yaratiladi va u class body tashqarisida accessible emas.

```
Private field access mexanizmi:

  class User { #name; }

  1. Class parse → PrivateName { description: "#name" } yaratiladi
  2. Instance yaratilganda → internal slot [[PrivateBrand]] o'rnatiladi
  3. this.#name access → Brand Check:
     - instance.[[PrivateBrand]] === User ? → OK, qiymat qaytariladi
     - boshqa class instance? → TypeError: brand mismatch
  4. Class tashqarisidan #name → SyntaxError (parse vaqtida, runtime emas)
```

Brand check — bu runtime enforcement. Agar bir class ning instance'ini boshqa class ning private field'iga kiritmoqchi bo'lsangiz, `TypeError` tashlanadi. Bu WeakMap pattern dan farqli — WeakMap da enforcement faqat closure scope orqali, `#` fields da esa spec-level brand check mavjud.

V8 internally private field'larni object ning **Hidden Class** (V8 source kodida `Map` deb nomlanadi — lekin bu JavaScript'dagi `Map` data structure'dan butunlay boshqa tushuncha) ga maxsus private symbol sifatida qo'shadi. Bu oddiy property descriptor emas — `Object.getOwnPropertyNames()` va `Reflect.ownKeys()` ham ko'rmaydi. Closure-based privacy dan farqi: `#` fields instance'ning o'zida saqlanadi (memory-efficient), closure esa har bir instance uchun yangi function object yaratadi.

</details>

<details>
<summary><strong>Kod Misollari</strong></summary>

**Private fields `#` bilan** (zamonaviy usul):

```javascript
class User {
  #password;
  #loginAttempts = 0;

  constructor(name, password) {
    this.name = name;
    this.#password = this.#hash(password);
  }

  #hash(str) {
    // Private method — tashqaridan chaqirib bo'lmaydi
    return btoa(str); // (sodda misol)
  }

  authenticate(password) {
    this.#loginAttempts++;
    if (this.#loginAttempts > 5) {
      throw new Error("Urinishlar soni oshdi");
    }
    return this.#hash(password) === this.#password;
  }
}

const u = new User("Ali", "secret123");
console.log(u.authenticate("secret123")); // true
// u.#password;     // ❌ SyntaxError
// u.#hash("test"); // ❌ SyntaxError
```

**WeakMap pattern** (class tashqarisida saqlash):

```javascript
const _private = new WeakMap();

class Wallet {
  constructor(owner, balance) {
    _private.set(this, { balance, pin: null });
    this.owner = owner;
  }

  setPin(pin) {
    _private.get(this).pin = pin;
  }

  getBalance(pin) {
    const data = _private.get(this);
    if (data.pin !== pin) throw new Error("Noto'g'ri PIN");
    return data.balance;
  }
}

const wallet = new Wallet("Ali", 500_000);
wallet.setPin(1234);
console.log(wallet.getBalance(1234)); // 500000
// Private ma'lumotlarga bevosita kirib bo'lmaydi
// WeakMap kaliti = this, GC instance o'chirilganda tozalaydi
```

### 2. Abstraction

#### Nazariya

Abstraction — murakkab ichki tizimni **yashirish** va foydalanuvchiga faqat **kerakli qismini** ko'rsatish. Foydalanuvchi "nima qiladi" ni biladi, "qanday qiladi" ni bilmasligi kerak.

JavaScript da `abstract` keyword yo'q (Java/C# dan farqli). Lekin abstraction ni bir necha pattern orqali qilish mumkin:

```
┌──────────────────────────────────────────────────────────┐
│          JS DA ABSTRACTION PATTERN'LARI                  │
├───────────────────────────┬──────────────────────────────┤
│ 1. Abstract class pattern │ throw Error + new.target     │
│ 2. Interface emulation    │ Duck typing / method check   │
│ 3. Facade pattern         │ Murakkab API ni soddalash    │
│ 4. Public API + private   │ Faqat kerakli method expose  │
└───────────────────────────┴──────────────────────────────┘
```

<details>
<summary><strong>Under the Hood</strong></summary>

`new.target` — ECMAScript spec dagi **NewTarget** meta-property. Bu constructor function ichida `[[Construct]]` internal method orqali beriladi. `new.target` ning ishlash mexanizmi:

```
new Child()  chaqirilganda:

  1. [[Construct]](args, newTarget) ishga tushadi
     - newTarget = Child (new dan keyingi constructor)
  2. super() chaqirilsa → parent.[[Construct]](args, newTarget)
     - newTarget hali ham Child — parent ga forwarding qilinadi
  3. Parent constructor ichida:
     - new.target === Child (Parent emas!)
     - if (new.target === Parent) throw Error → ISHLAMAYDI
     - Chunki new.target = Child
  4. new Parent() to'g'ridan chaqirilsa:
     - new.target === Parent → throw Error ISHLAYDI
```

`[[Construct]]` va `[[Call]]` — function object ning ikki xil internal method. Oddiy chaqiruv `fn()` `[[Call]]` ni ishlatadi, `new fn()` esa `[[Construct]]` ni. Arrow function'larda `[[Construct]]` yo'q — shu sababli `new` bilan chaqirib bo'lmaydi. `new.target` faqat `[[Construct]]` kontekstida mavjud, `[[Call]]` da `undefined` bo'ladi.

V8 da `new.target` check juda arzon — bu oddiy register comparison. Abstract class pattern'i runtime cost deyarli qo'shmaydi, chunki check faqat constructor ichida bir marta bajariladi.

</details>

<details>
<summary><strong>Kod Misollari</strong></summary>

**Abstract class pattern:**

```javascript
class Shape {
  constructor() {
    if (new.target === Shape) {
      throw new Error("Shape abstract class — to'g'ridan yaratib bo'lmaydi");
    }
  }

  // "Abstract" method — subclass Override qilishi SHART
  area() {
    throw new Error(`${this.constructor.name} da area() implement qilinmagan`);
  }

  // "Abstract" method
  perimeter() {
    throw new Error(`${this.constructor.name} da perimeter() implement qilinmagan`);
  }

  // Concrete method — barchaga umumiy
  describe() {
    return `${this.constructor.name}: area=${this.area()}, perimeter=${this.perimeter()}`;
  }
}

class Circle extends Shape {
  constructor(radius) {
    super();
    this.radius = radius;
  }

  area() {
    return Math.PI * this.radius ** 2;
  }

  perimeter() {
    return 2 * Math.PI * this.radius;
  }
}

class Rectangle extends Shape {
  constructor(width, height) {
    super();
    this.width = width;
    this.height = height;
  }

  area() {
    return this.width * this.height;
  }

  perimeter() {
    return 2 * (this.width + this.height);
  }
}

// const s = new Shape(); // ❌ Error: abstract class
const c = new Circle(5);
console.log(c.describe()); // "Circle: area=78.53981633974483, perimeter=31.41592653589793"

const r = new Rectangle(4, 6);
console.log(r.describe()); // "Rectangle: area=24, perimeter=20"
```

**Facade pattern — murakkablikni yashirish:**

```javascript
// Murakkab ichki tizim:
class VideoDecoder {
  decode(file) { return `${file} decoded`; }
}
class AudioDecoder {
  decode(file) { return `${file} audio extracted`; }
}
class SubtitleParser {
  parse(file) { return `${file} subtitles parsed`; }
}
class Renderer {
  render(video, audio, subtitles) {
    return `Playing: ${video} + ${audio} + ${subtitles}`;
  }
}

// Facade — sodda interface:
class MediaPlayer {
  #video = new VideoDecoder();
  #audio = new AudioDecoder();
  #subtitle = new SubtitleParser();
  #renderer = new Renderer();

  play(file) {
    // Foydalanuvchi faqat play() ni bilsa bas
    const v = this.#video.decode(file);
    const a = this.#audio.decode(file);
    const s = this.#subtitle.parse(file);
    return this.#renderer.render(v, a, s);
  }
}

const player = new MediaPlayer();
console.log(player.play("movie.mp4"));
// Foydalanuvchi ichki decoder/parser/renderer haqida bilmaydi
```

</details>

### 3. Inheritance

#### Nazariya

Inheritance — bir class (child/subclass) boshqa class (parent/superclass) dan xususiyat va method'larni **meros olish** mexanizmi. Bu kodni qayta ishlatish (code reuse) imkonini beradi.

JavaScript dagi inheritance **prototypal** — ya'ni ob'ektlar boshqa ob'ektlardan bevosita meros oladi. ES6 `class` + `extends` bu mexanizmga qulay sintaksis beradi.

> **Batafsil:** Prototypal inheritance → [07-prototypes.md](07-prototypes.md), Classes inheritance → [08-classes.md](08-classes.md)

```
┌─────────────────────────────────────────────────────┐
│         JS DA INHERITANCE USULLARI                  │
├─────────────────────────┬───────────────────────────┤
│ Prototypal              │ Object.create()           │
│ Constructor + prototype │ function + new            │
│ ES6 Class extends       │ class Child extends Parent│
│ Mixins                  │ Object.assign / spread    │
│ Composition             │ has-a (inheritance o'rniga)│
└─────────────────────────┴───────────────────────────┘
```

<details>
<summary><strong>Under the Hood</strong></summary>

`extends` keyword spec da **prototype chain linking** ni amalga oshiradi. `class Child extends Parent` yozilganda engine ichida ikki darajali link o'rnatiladi:

```
class Parent { }
class Child extends Parent { }

Engine ichida:
  1. Child.prototype.[[Prototype]] = Parent.prototype
     → instance method meros (eat, sleep)
  2. Child.[[Prototype]] = Parent
     → static method meros (Parent.staticMethod)

  Object.getPrototypeOf(Child.prototype) === Parent.prototype  // true
  Object.getPrototypeOf(Child) === Parent                      // true
```

`Object.create(proto)` spec da `OrdinaryObjectCreate(proto)` ni chaqiradi — yangi ob'ekt yaratib, uning `[[Prototype]]` internal slot'ini `proto` ga o'rnatadi. Bu `new` operator dan farqli — constructor chaqirilmaydi, faqat prototype link o'rnatiladi.

`super()` chaqiruvi spec da `[[Construct]]` ni parent constructor ga `newTarget` ni forwarding qilib delegate qiladi. `super()` dan oldin child constructor da `this` ga murojaat qilish `ReferenceError` beradi — chunki `this` binding faqat `super()` dan keyin **TDZ** (Temporal Dead Zone) dan chiqadi. Bu spec da "derived constructor" maxsus holati — base constructor'lar esa `this` ni avtomatik oladi.

V8 da prototype chain lookup **inline cache** (IC) bilan optimizatsiya qilinadi — birinchi marta property qidirilganda chain bo'ylab yuradi, keyin natija cache'lanadi. Shu sababli `buddy.eat()` birinchi chaqiruvdan keyin deyarli to'g'ridan-to'g'ri (direct) access tezligida ishlaydi.

</details>

<details>
<summary><strong>Kod Misollari</strong></summary>

**Inheritance zanjiri:**

```javascript
class Animal {
  constructor(name) {
    this.name = name;
    this.isAlive = true;
  }

  eat(food) {
    return `${this.name} ${food} yeyapti`;
  }

  sleep() {
    return `${this.name} uxlayapti`;
  }
}

class Dog extends Animal {
  constructor(name, breed) {
    super(name);
    this.breed = breed;
  }

  bark() {
    return `${this.name} havlayapti!`;
  }
}

class GuideDog extends Dog {
  constructor(name, breed, owner) {
    super(name, breed);
    this.owner = owner;
  }

  guide() {
    return `${this.name} ${this.owner}ni boshqarayapti`;
  }
}

const buddy = new GuideDog("Buddy", "Labrador", "Ali");
console.log(buddy.eat("go'sht"));  // "Buddy go'sht yeyapti" — Animal dan
console.log(buddy.bark());         // "Buddy havlayapti!" — Dog dan
console.log(buddy.guide());        // "Buddy Alini boshqarayapti" — o'ziniki
console.log(buddy instanceof GuideDog); // true
console.log(buddy instanceof Animal);   // true
```

```
Prototype Chain:
buddy → GuideDog.prototype → Dog.prototype → Animal.prototype → Object.prototype → null
```

</details>

### 4. Polymorphism

#### Nazariya

Polymorphism — "ko'p shakllilik". Bir xil nomli method turli class'larda **turli xil** ishlashi. Bu OOP ning eng kuchli ustunlaridan biri — kodni umumiy (generic) qilib yozish imkonini beradi.

JS da polymorphism ning 3 turi mavjud:

```
┌──────────────────────────────────────────────────────────────┐
│                 JS DA POLYMORPHISM TURLARI                   │
├────────────────────┬─────────────────────────────────────────┤
│ Method Overriding  │ Child class parent method ni qayta      │
│                    │ yozadi (eng keng tarqalgan)              │
├────────────────────┼─────────────────────────────────────────┤
│ Duck Typing        │ "Agar o'rdakdek yursa, o'rdakdek        │
│                    │ qaqillasa — u o'rdak". Type emas,       │
│                    │ behavior muhim.                          │
├────────────────────┼─────────────────────────────────────────┤
│ Method Overloading │ Bir xil nom, turli parametrlar.         │
│ (emulation)        │ JS da native yo'q — shart bilan         │
│                    │ emulatsiya qilinadi.                     │
└────────────────────┴─────────────────────────────────────────┘
```

> **Muhim:** JS da **operator overloading** yo'q (C++ dan farqli). Lekin `Symbol.toPrimitive`, `valueOf`, `toString` orqali type coercion xatti-harakatini o'zgartirish mumkin (17-bo'limda batafsil).

<details>
<summary><strong>Under the Hood</strong></summary>

Polymorphism V8 da **dynamic dispatch** orqali ishlaydi. `shape.area()` chaqirilganda engine quyidagi method resolution tartibini bajaradi:

```
shape.area() chaqiruvi:

  1. shape ob'ektining Hidden Class da "area" qidiriladi
  2. Topilsa → shu method chaqiriladi (own property)
  3. Topilmasa → shape.[[Prototype]] da qidiriladi
  4. Prototype chain bo'ylab yuqoriga qarab davom etadi
  5. Birinchi topilgan "area" method chaqiriladi

  Circle instance:  area → Circle.prototype.area ✓ (1-step)
  Triangle instance: area → Triangle.prototype.area ✓ (1-step)
  Shape instance:    area → Shape.prototype.area ✓ (1-step, throws)
```

> **Eslatma:** "Hidden Class" V8 source kodida `Map` deb nomlanadi (JavaScript'ning `Map` data structure'idan farqli — bu engine ichki strukturasi).

V8 **inline caching (IC)** orqali polymorphic call'larni optimizatsiya qiladi. IC ning 3 holati bor:
- **Monomorphic** — faqat bitta type ko'rilgan (eng tez, direct call)
- **Polymorphic** — 2-4 ta type ko'rilgan (cache array, tez)
- **Megamorphic** — 5+ type (generic lookup, sekinroq)

`shapes.forEach(s => s.area())` da har xil Shape type'lar berilsa, IC polymorphic holatga o'tadi. V8 har bir ko'rilgan hidden class uchun alohida cache entry saqlaydi. Hidden class o'zgarganda (masalan, property qo'shilsa) cache invalidate bo'ladi.

Duck typing JS da type check yo'qligi sababli ishlaydi — engine method mavjudligini faqat property lookup orqali tekshiradi. `typeof obj.speak === "function"` — bu oddiy property access + type comparison, hech qanday class hierarchy talab qilinmaydi.

</details>

<details>
<summary><strong>Kod Misollari</strong></summary>

**Method Overriding — eng asosiy polymorphism:**

```javascript
class Shape {
  area() {
    return 0;
  }

  toString() {
    return `${this.constructor.name}: area = ${this.area().toFixed(2)}`;
  }
}

class Circle extends Shape {
  constructor(r) { super(); this.r = r; }
  area() { return Math.PI * this.r ** 2; }
}

class Triangle extends Shape {
  constructor(base, height) { super(); this.base = base; this.height = height; }
  area() { return 0.5 * this.base * this.height; }
}

class Square extends Shape {
  constructor(side) { super(); this.side = side; }
  area() { return this.side ** 2; }
}

// ✅ Polymorphism kuchi — bir xil method, turli natija:
const shapes = [new Circle(5), new Triangle(4, 6), new Square(3)];

for (const shape of shapes) {
  console.log(shape.toString());
}
// "Circle: area = 78.54"
// "Triangle: area = 12.00"
// "Square: area = 9.00"

// ✅ Funktsiya barcha Shape'lar bilan ishlaydi — type bilishi shart emas:
function totalArea(shapes) {
  return shapes.reduce((sum, s) => sum + s.area(), 0);
}
console.log(totalArea(shapes)); // 99.53981633974483
```

**Duck Typing — JS ning kuchli tarafi:**

```javascript
// Bu ob'ektlar hech qanday umumiy class'dan meros olmagan
// Lekin hammasi "speak" va "name" ga ega — shu yetarli

const dog  = { name: "Rex",   speak() { return "Hav!"; } };
const cat  = { name: "Murka", speak() { return "Miyau!"; } };
const duck = { name: "Donald",speak() { return "Quack!"; } };

// ✅ Duck typing — type emas, behavior muhim:
function makeThemSpeak(animals) {
  for (const animal of animals) {
    // animal.speak() mavjudligini tekshiryapmiz, type ni emas
    console.log(`${animal.name}: ${animal.speak()}`);
  }
}

makeThemSpeak([dog, cat, duck]);
// "Rex: Hav!"
// "Murka: Miyau!"
// "Donald: Quack!"

// ✅ Real-world misol — iterable protocol:
// Array, String, Map, Set — hammasi Symbol.iterator ga ega
// for...of hammasi bilan ishlaydi — type muhim emas, protocol muhim
for (const x of [1, 2]) {}       // Array
for (const x of "abc") {}        // String
for (const x of new Set([1])) {} // Set
// Hammasi "iterable" — duck typing!
```

**Method Overloading emulatsiyasi:**

## Prototypal vs Classical OOP

### Nazariya

Dasturlash tillarida OOP ikki xil tizimda ishlaydi:

| | **Classical OOP** (Java, C#, C++) | **Prototypal OOP** (JavaScript) |
|---|---|---|
| **Meros mexanizmi** | Class dan class ga (class → instance) | Ob'ektdan ob'ektga (`[[Prototype]]` chain) |
| **Tushuncha** | Class — blueprint (shablon) | Prototype — tirik ob'ekt |
| **Abstract class** | `abstract` keyword — til darajasida | Yo'q — pattern bilan emulatsiya |
| **Interface** | `interface` keyword | Yo'q — duck typing |
| **Multiple inheritance** | Ko'pchilik taqiqlaydi (Java) | Mixins bilan mumkin |
| **Instance yaratish** | `new ClassName()` | `Object.create()`, `new`, literal |
| **Runtime o'zgartirish** | Mumkin emas (class qotib qoladi) | Istalgan vaqtda prototype o'zgaradi |
| **Flexible** | Kam | Juda yuqori |

```
Classical OOP:                   Prototypal OOP:

  ┌─────────┐                    ┌─────────────┐
  │  Class   │ (blueprint)       │  Prototype   │ (tirik ob'ekt)
  │ (shablon)│                   │  Object      │
  └────┬─────┘                   └──────┬───────┘
       │ new                            │ [[Prototype]]
       ▼                                ▼
  ┌─────────┐                    ┌─────────────┐
  │ Instance │ (nusxa)           │   Object     │ (prototype'dan meros)
  └──────────┘                   └──────────────┘
```

<details>
<summary><strong>Under the Hood</strong></summary>

Classical OOP (Java, C++) da class — compile-time construct. Instance yaratilganda class layout asosida memory allocate qilinadi va method'lar **vtable** (virtual method table) orqali dispatch qilinadi. Class o'zgarmas — runtime da yangi method qo'shib bo'lmaydi.

JS da class — runtime object. `class Vehicle {}` yozilganda engine ikkita ob'ekt yaratadi: `Vehicle` (constructor function) va `Vehicle.prototype` (method holder). `extends` bu ob'ektlar orasida `[[Prototype]]` link o'rnatadi. Bu link runtime da o'zgarishi mumkin — `Object.setPrototypeOf()` bilan.

```
Classical (Java):                    JS Prototypal:

  ┌──────────┐                       ┌──────────────┐
  │  Class    │ (compile-time)       │  Vehicle     │ (function object)
  │  vtable[] │                      │  .prototype ─┼──→ { start(), stop() }
  └──────────┘                       └──────────────┘
       │ fixed layout                     │ [[Prototype]] = dynamic
       ▼                                  ▼
  ┌──────────┐                       ┌──────────────┐
  │ Instance  │ (vtable pointer)     │  car object  │ (own props + proto link)
  └──────────┘                       └──────────────┘
```

V8 da `Object.create(vehicle)` spec ning `OrdinaryObjectCreate` ni chaqiradi — yangi ob'ekt yaratib `[[Prototype]]` ni `vehicle` ga bog'laydi. Runtime da `vehicle` ga yangi method qo'shilsa, `car` darhol ko'radi — chunki link tirik reference, nusxa emas.

</details>

<details>
<summary><strong>Kod Misollari</strong></summary>

```javascript
// ✅ Prototypal OOP — ob'ektdan ob'ektga bevosita meros:
const vehicle = {
  start() { return `${this.name} yurdi`; },
  stop()  { return `${this.name} to'xtadi`; }
};

const car = Object.create(vehicle);
car.name = "Tesla";
car.honk = function() { return "Beep!"; };

console.log(car.start()); // "Tesla yurdi" — vehicle dan meros
console.log(car.honk());  // "Beep!" — o'ziniki

// Runtime da prototype o'zgartirish mumkin:
vehicle.fuel = function() { return "yonilg'i to'ldirildi"; };
console.log(car.fuel()); // "yonilg'i to'ldirildi" — darhol ishlaydi!

// ✅ ES6 class — prototype ustiga qurilgan qulay sintaksis:
class Vehicle {
  constructor(name) { this.name = name; }
  start() { return `${this.name} yurdi`; }
}

// Ichida xuddi shu prototype mexanizmi ishlaydi
```

</details>

## Delegation Pattern

### Nazariya

Delegation — ob'ekt o'zi bajara olmaydigan ishni boshqa ob'ektga **topshirish**. JavaScript'ning prototypal inheritance aslida delegation'ga asoslangan — ob'ektda method topilmasa, prototype chain bo'ylab **delegatsiya** qilinadi.

Delegation inheritance'dan farqli — bu "is-a" emas, balki "**handles-by-forwarding**" munosabati:

```
Inheritance:                    Delegation:
  Parent                          ┌──────────┐
    ↑ is-a                        │ Object A │
  Child                           │ method() │──→ delegates to ──→ Object B
  (method o'z ichida)             └──────────┘                    (method shu yerda)
```

<details>
<summary><strong>Under the Hood</strong></summary>

OLOO pattern da `Object.create(Validator)` chaqirilganda spec ning `OrdinaryObjectCreate(Validator)` ishlaydi — yangi ob'ekt yaratiladi va uning `[[Prototype]]` internal slot'i `Validator` ga point qiladi. Method call `form.validate()` da engine quyidagi lookup ni bajaradi:

```
form.validate() call:

  1. form own properties da "validate" qidiriladi → yo'q
  2. form.[[Prototype]] = UserForm da qidiriladi → yo'q
  3. UserForm.[[Prototype]] = Validator da qidiriladi → TOPILDI
  4. Validator.validate() chaqiriladi, this = form

  form → UserForm → Validator
  [[Prototype]]  [[Prototype]]

  Muhim: this binding delegation da HAM ishlaydi.
  validate() ichida this.rules → form.rules (form ning o'zi)
```

Delegation inheritance dan farqi internal darajada: inheritance da child ob'ekt parent ning method'larini **o'z prototype chain** ida ko'radi, delegation da esa ob'ekt method'ni bajara olmaganida **forwarding** qiladi. Lekin JS da ikkalasi ham aynan bir xil `[[Prototype]]` lookup mechanism orqali ishlaydi — farq kontseptual, implementation bir xil.

V8 da prototype chain lookup birinchi marta `form.validate()` chaqirilganda to'liq chain bo'ylab yuradi, keyin inline cache (IC) orqali natija saqlanadi. Keyingi chaqiruvlarda lookup skip qilinadi — to'g'ridan-to'g'ri cached function chaqiriladi. Bu delegation pattern ning performance cost'ini deyarli nolga tushiradi.

</details>

<details>
<summary><strong>Kod Misollari</strong></summary>

**OLOO — Objects Linking to Other Objects (Kyle Simpson pattern):**

```javascript
// ✅ Delegation — class'siz, ob'ektlar bir-biriga ulangan:
const Validator = {
  validate() {
    for (const [key, rule] of Object.entries(this.rules || {})) {
      if (rule === "required" && !this[key]) {
        throw new Error(`${key} majburiy`);
      }
    }
    return true;
  }
};

const UserForm = Object.create(Validator);
UserForm.init = function(name, email) {
  this.name = name;
  this.email = email;
  this.rules = { name: "required", email: "required" };
  return this;
};

UserForm.submit = function() {
  this.validate(); // Validator ga delegatsiya
  return `${this.name} submitted`;
};

const form = Object.create(UserForm);
form.init("Ali", "ali@mail.com");
console.log(form.submit()); // "Ali submitted"
// form → UserForm → Validator — delegation chain
```

**Explicit delegation (forwarding):**

```javascript
class Logger {
  log(msg) { console.log(`[LOG] ${msg}`); }
  warn(msg) { console.warn(`[WARN] ${msg}`); }
  error(msg) { console.error(`[ERROR] ${msg}`); }
}

class UserService {
  #logger;
  #users = [];

  constructor(logger) {
    this.#logger = logger; // delegation — Logger'ga topshiradi
  }

  addUser(name) {
    this.#users.push(name);
    this.#logger.log(`User qo'shildi: ${name}`); // delegatsiya
  }

  removeUser(name) {
    const idx = this.#users.indexOf(name);
    if (idx === -1) {
      this.#logger.warn(`User topilmadi: ${name}`); // delegatsiya
      return false;
    }
    this.#users.splice(idx, 1);
    this.#logger.log(`User o'chirildi: ${name}`);
    return true;
  }
}

const service = new UserService(new Logger());
service.addUser("Ali"); // [LOG] User qo'shildi: Ali
// UserService log qilishni BILMAYDI — Logger'ga TOPSHIRADI
```

**Delegation vs Inheritance farqi:**

| | **Inheritance** | **Delegation** |
|---|---|---|
| **Munosabat** | is-a (Child is a Parent) | has-a / uses-a |
| **Bog'liqlik** | Qattiq — compile-time | Sust — runtime o'zgarishi mumkin |
| **Flexibility** | Kam — hierarxiya qotib qoladi | Yuqori — delegat almashtiriladi |
| **JS'da** | `extends`, prototype chain | Object.create, composition, forwarding |

</details>

## Method Chaining — Fluent Interface

### Nazariya

Method chaining — har bir method `this` ni qaytaradi, shunda bir necha method'ni **zanjir** shaklida chaqirish mumkin bo'ladi. Bu pattern "Fluent Interface" deb ataladi va ko'plab JS kutubxonalarida ishlatiladi (jQuery, Lodash, Knex, D3.js).

```
Oddiy chaqiruv:                  Method Chaining:

builder.setName("Ali");          builder
builder.setAge(25);                .setName("Ali")
builder.setEmail("a@b.com");       .setAge(25)
const user = builder.build();      .setEmail("a@b.com")
                                   .build();
```

<details>
<summary><strong>Under the Hood</strong></summary>

`return this` — V8 da juda yaxshi optimizatsiya qilinadigan pattern. Har bir method chaqiruvi `this` reference'ni qaytarganda, engine buning **yangi ob'ekt yaratish emasligini** biladi va allocation skip qiladi.

```
builder.from("users").select("name").where("age > 18")

V8 execution:
  1. builder.from("users")
     - this = builder (receiver)
     - hidden class: Map0 { #table, #conditions, ... }
     - return this → xuddi shu ob'ekt reference qaytadi
     - IC: from() → monomorphic (bitta type)

  2. .select("name")
     - this = builder (AYNAN SHU ob'ekt)
     - hidden class O'ZGARMAGAN → IC hit (tez)
     - return this → xuddi shu reference

  3. .where("age > 18")
     - this = builder → IC hit → tez
```

V8 ning inline cache (IC) tizimi method chaining uchun ideal: zanjirdagi har bir method call aynan bitta ob'ekt type'ida ishlaydi (monomorphic). Hidden class o'zgarmaydi (chunki `#table`, `#conditions` constructor da yaratilgan), shu sababli IC doimiy hit beradi — bu eng tez dispatch holati.

Immutable chaining (`new Vector(...)` qaytarish) da esa har bir chaqiruvda yangi ob'ekt allocate bo'ladi. V8 ning **Escape Analysis** agar ob'ekt method tashqarisiga "chiqmasa" heap allocation o'rniga stack allocation qilishi mumkin, lekin chaining da ob'ekt qaytarilgani uchun bu optimizatsiya kamroq qo'llaniladi.

</details>

<details>
<summary><strong>Kod Misollari</strong></summary>

**Asosiy pattern — `return this`:**

```javascript
class QueryBuilder {
  #table = "";
  #conditions = [];
  #columns = ["*"];
  #orderBy = "";
  #limit = null;

  from(table) {
    this.#table = table;
    return this; // ← zanjir uchun this qaytariladi
  }

  select(...columns) {
    this.#columns = columns;
    return this;
  }

  where(condition) {
    this.#conditions.push(condition);
    return this;
  }

  order(column, dir = "ASC") {
    this.#orderBy = `ORDER BY ${column} ${dir}`;
    return this;
  }

  take(n) {
    this.#limit = n;
    return this;
  }

  build() {
    const cols = this.#columns.join(", ");
    const where = this.#conditions.length
      ? `WHERE ${this.#conditions.join(" AND ")}`
      : "";
    const limit = this.#limit ? `LIMIT ${this.#limit}` : "";
    return `SELECT ${cols} FROM ${this.#table} ${where} ${this.#orderBy} ${limit}`.trim();
  }
}

// ✅ Fluent Interface — o'qish oson, yozish qulay:

  .select("name", "email", "age")

```

## Symbol.hasInstance — Custom instanceof

### Nazariya

`instanceof` operator ichida `Symbol.hasInstance` well-known symbol ishlatiladi. Uni override qilib, `instanceof` ning xatti-harakatini **boshqarish** mumkin — bu polymorphism'ning metaprogramming darajasi.

> **Havola:** Symbol'lar haqida batafsil → [17-type-coercion.md](17-type-coercion.md)

<details>
<summary><strong>Under the Hood</strong></summary>

`instanceof` operator spec da **InstanceofOperator** abstract operation ni chaqiradi. Bu operation avval `Symbol.hasInstance` method'ni qidiradi, topilmasa **OrdinaryHasInstance** ga fallback qiladi:

```
obj instanceof Constructor:

  1. hasInstance = Constructor[Symbol.hasInstance] qidiriladi
  2. Agar topilsa → hasInstance.call(Constructor, obj) chaqiriladi
     - return value Boolean ga convert qilinadi
  3. Agar topilmasa → OrdinaryHasInstance(Constructor, obj):
     a. proto = Constructor.prototype
     b. obj.[[Prototype]] chain bo'ylab proto qidiriladi
     c. Topilsa → true, null ga yetilsa → false
```

`Symbol.hasInstance` — **well-known symbol** lardan biri. Well-known symbol'lar engine ning internal behavior'ini override qilish uchun mo'ljallangan: `Symbol.iterator`, `Symbol.toPrimitive`, `Symbol.hasInstance`, `Symbol.species` va boshqalar. Bular global `Symbol` registr'ida saqlanadi va `Symbol.hasInstance === Symbol.hasInstance` doim `true` (har safar yangi symbol yaratilmaydi).

`static [Symbol.hasInstance]()` yozilganda bu method **constructor function ning o'ziga** (prototype emas) qo'shiladi. `Function.prototype[Symbol.hasInstance]` default implementation — OrdinaryHasInstance ni chaqiradi. Static method uni **shadow** qiladi, shu sababli custom logic ishlaydi.

</details>

<details>
<summary><strong>Kod Misollari</strong></summary>

```javascript
// ✅ Custom instanceof — duck typing bilan birlashtirish:
class Iterable {
  static [Symbol.hasInstance](instance) {
    return instance != null && typeof instance[Symbol.iterator] === "function";
  }
}

console.log([] instanceof Iterable);           // true — Array iterable
console.log("hello" instanceof Iterable);      // true — String iterable
console.log(new Map() instanceof Iterable);    // true — Map iterable
console.log(new Set([1]) instanceof Iterable); // true — Set iterable
console.log(123 instanceof Iterable);          // false — number iterable emas
console.log({} instanceof Iterable);           // false — oddiy object emas

// ✅ Interface-like type checking:
class Serializable {
  static [Symbol.hasInstance](instance) {
    return instance != null
      && typeof instance.toJSON === "function"
      && typeof instance.toString === "function";
  }
}

class User {
  constructor(name) { this.name = name; }
  toJSON() { return { name: this.name }; }
  toString() { return this.name; }
}

console.log(new User("Ali") instanceof Serializable); // true
console.log({} instanceof Serializable);               // false

// ✅ Version checking:
class ModernBrowser {
  static [Symbol.hasInstance](instance) {
    return typeof instance.fetch === "function"
      && typeof instance.Promise !== "undefined"
      && typeof instance.Proxy !== "undefined";
  }
}

// Browser environment da:
// console.log(window instanceof ModernBrowser); // true/false
```

</details>

## SOLID Printsiplari

### Nazariya

SOLID — 5 ta OOP dizayn printsipini bildiradi. Robert C. Martin (Uncle Bob) tomonidan tizimlashtirilgan. Bu printsiplar kodni **maintainable**, **flexible** va **scalable** qilishga yordam beradi.

```
┌───────────────────────────────────────────────────────────────┐
│                      SOLID                                    │
├───┬───────────────────────┬───────────────────────────────────┤
│ S │ Single Responsibility │ Har bir class bitta vazifa        │
│ O │ Open/Closed           │ Kengaytirishga ochiq,             │
│   │                       │ o'zgartirishga yopiq              │
│ L │ Liskov Substitution   │ Child parent o'rnida ishlashi     │
│   │                       │ kerak                             │
│ I │ Interface Segregation  │ Katta interface → kichiklarga    │
│ D │ Dependency Inversion  │ Abstraktsiyaga bog'lan,           │
│   │                       │ konkret class ga emas             │
└───┴───────────────────────┴───────────────────────────────────┘
```

<details>
<summary><strong>Under the Hood</strong></summary>

SOLID printsiplari asli **statik typed** tillar (C++, Java, C#) uchun Robert C. Martin tomonidan tizimlashtirilgan. JavaScript'ning **dinamik va prototypal** tabiati har bir printsipga o'ziga xos xususiyat qo'shadi — ular klassik OOP tillaridagi SOLID'dan farqli amalda qo'llaniladi. Quyidagi bo'lim har bir SOLID printsipining **JavaScript kontekstida** qanday amalda qo'llanilishini va engine darajasidagi aloqasini yoritadi:

**S — Single Responsibility**

JS'da SRP modul darajasida ham aniq ko'rinadi. ES Modules har bir fayl alohida **Module Record** — faqat bitta vazifaga mas'ul modul **tree-shaking** (dead code elimination) da yaxshi optimize bo'ladi. V8 ning static analysis low cohesion (ko'p responsibility'li) module'larda samarasiz ishlaydi — chunki bog'liq bo'lmagan funksiyalar bitta module'da bo'lsa, bundler qaysi birini olib tashlash mumkinligini aniqlay olmaydi. SRP nafaqat class darajasida, modul darajasida ham qo'llanishi kerak.

**O — Open/Closed**

Prototypal OOP OCP ni **runtime darajasida** ta'minlaydi — `Class.prototype.newMethod = ...` bilan mavjud kodni o'zgartirmasdan class'ni kengaytirish mumkin. Statik tillarda bu mumkin emas (class compile-time'da qotadi). Lekin runtime extension'ning **sharti** — V8 Hidden Class va inline cache invalidation: mavjud instance'lar yangi method'ni ko'radi, lekin IC cache qayta qurilishi kerak. Shu sababli OCP ni "runtime plugin" sifatida emas, **class definition vaqtida** (subclassing, composition) qo'llash ko'proq optimize.

**L — Liskov Substitution**

JS **duck typing** LSP ga ikki tomonlama ta'sir qiladi. Bir tomondan — nominal type system yo'qligi formal LSP tekshirishni qiyinlashtiradi (TypeScript bilan yaxshilanadi). Ikkinchi tomondan — duck typing LSP'ning asosini allaqachon o'zida saqlaydi: agar method mavjud bo'lsa va to'g'ri signature'ga ega bo'lsa, u ishlaydi — rasmiy type hierarchy shart emas. Ya'ni JavaScript'da "behavioral substitutability" prinsipi tip munosabat'idan ko'ra muhim.

**I — Interface Segregation**

JS'da `interface` keyword yo'q. ISP **composition** va **mixin** pattern'lari orqali amalga oshiriladi — har "interface" factory function yoki kichik capability set yaratadi. `Symbol.hasInstance` bilan duck-typing interface check ham mumkin (faylning yuqorisida Iterable interface misoli). TypeScript interface'lar **compile-time'da yo'qoladi** — runtime'da JS OOP qoidalari amal qiladi, shuning uchun `instanceof` runtime interface check uchun ishlamaydi. Bu faqat structural check orqali qilinishi kerak.

**D — Dependency Inversion**

JS'da **DI konteynerlar shart emas** — constructor injection va closure parametrlari kifoya. Dinamik typing abstraksiyani **implicit** qiladi: `new Service(logger)` — `logger` parametri aniq tip talab qilmaydi, faqat expected method'larga ega bo'lishi kerak (duck typing). Bu Inversion of Control (IoC) container'larsiz amalga oshirish imkonini beradi — React, Express, Node.js kutubxonalarining aksariyati shu yondashuvdan foydalanadi. DI container'lar JS'da asosan Angular, NestJS kabi strict arxitekturalarda uchraydi.

**Xulosa:** Har bir SOLID principle JavaScript kontekstida **yumshroq** — rigid type system yo'qligi flexibility beradi, lekin discipline ko'proq kerak. Compile-time tekshirishlar yo'q, shu sababli design violation'lar runtime'da yoki testlar orqali topiladi. TypeScript ko'p muammolarni compile-time'ga olib keladi, lekin runtime semantics hali ham JS'ning.

</details>

### S — Single Responsibility

Har bir class **bitta vazifa** uchun javobgar bo'lishi kerak. Agar class'ni o'zgartirish uchun birdan ortiq sabab bo'lsa — uni bo'lish kerak.

```javascript
// ❌ SRP buzilgan — bitta class juda ko'p vazifa:
class User {
  constructor(name, email) {
    this.name = name;
    this.email = email;
  }

  save() {
    // DB ga saqlash — User ning vazifasi emas
    db.insert("users", { name: this.name, email: this.email });
  }

  sendEmail(message) {
    // Email yuborish — User ning vazifasi emas
    mailer.send(this.email, message);
  }

  generateReport() {
    // Hisobot yaratish — User ning vazifasi emas
    return pdf.create(`Report for ${this.name}`);
  }
}


// ✅ SRP to'g'ri — har bir class bitta vazifa:
class User {
  constructor(name, email) {
    this.name = name;
    this.email = email;
  }
}
```

## Coupling va Cohesion

### Nazariya

OOP dizayning ikkita eng muhim o'lchovi — **Coupling** (bog'liqlik) va **Cohesion** (bog'langanlik). Yaxshi dizayn = **loose coupling** + **high cohesion**.

```
┌───────────────────────────────────────────────────────────────┐
│                COUPLING VA COHESION                            │
├─────────────────────────┬─────────────────────────────────────┤
│ Coupling (bog'liqlik)   │ Cohesion (bog'langanlik)            │
│                         │                                     │
│ Modullar o'rtasidagi    │ Modul ICHIDAGI elementlarning       │
│ bog'liqlik darajasi     │ bir-biriga aloqadorlik darajasi     │
│                         │                                     │
│ ❌ Tight = modullar     │ ❌ Low = modul ichida turli xil,   │
│    bir-biriga qattiq    │    bog'liq bo'lmagan vazifalar     │
│    bog'liq              │                                     │
│                         │                                     │
│ ✅ Loose = modullar     │ ✅ High = modul ichida faqat       │
│    mustaqil, alohida    │    bir-biriga bog'liq vazifalar    │
│    o'zgaradi            │                                     │
└─────────────────────────┴─────────────────────────────────────┘
```

<details>
<summary><strong>Under the Hood</strong></summary>

Coupling va cohesion JavaScript module system internals bilan bevosita bog'liq. ES Modules da har bir fayl alohida **Module Record** bo'ladi. `import` statement spec da **ImportEntry** record yaratadi — bu modullar orasidagi coupling ni formal qilib ko'rsatadi.

```
Tight coupling (module darajasida):

  // orderService.js
  import { db, query, TABLE_NAME } from './userService.js';
  //       ↑    ↑        ↑
  //    3 ta ImportEntry = tight coupling (ichki tuzilishga bog'liq)

Loose coupling:

  // orderService.js
  import { findById, canPurchase } from './userService.js';
  //         ↑           ↑
  //    2 ta public API method = loose coupling
```

V8 da module **static analysis** vaqtida (parse phase) barcha `import`/`export` binding'lar resolve qilinadi. Tight coupled module lar ko'p binding'ga ega — bitta module o'zgarganda boshqalari ham re-evaluate bo'lishi kerak. Loose coupled module'lar minimal binding orqali bog'langani uchun o'zgarishlar izolyatsiyalangan.

High cohesion module darajasida — bitta module ichidagi barcha export'lar bir-biriga semantik jihatdan bog'liq. V8 ning **tree-shaking** (dead code elimination) low cohesion module'larda samarasiz ishlaydi — chunki bog'liq bo'lmagan funksiyalar bitta module da bo'lsa, bundler qaysi birini olib tashlash mumkinligini aniqlashi qiyinlashadi.

</details>

## Law of Demeter

### Nazariya

Law of Demeter (LoD) yoki **Principle of Least Knowledge** — ob'ekt faqat o'zining **yaqin do'stlari** bilan gaplashishi kerak, "begonalar" bilan emas.

**Qoida:** Method faqat quyidagilar bilan muloqot qilishi mumkin:
1. O'z ob'ektining method/property'lari (`this`)
2. Method'ning parametrlari
3. Method ichida yaratilgan ob'ektlar
4. O'z ob'ektining tarkibiy qismlari (direct components)

**Buzilish belgisi:** Zanjirli dot notation — `a.b.c.d.doSomething()` (**"train wreck"**)

<details>
<summary><strong>Under the Hood</strong></summary>

Law of Demeter faqat design principle emas — V8 inline cache (IC) tizimiga ham bevosita ta'sir qiladi. Har bir `.` operator alohida **call site** hosil qiladi, va V8 har call site uchun alohida IC state saqlaydi:

```
order.customer.address.city

V8 nuqtai nazaridan — 3 ta alohida call site:

  call_site_1: order.customer
    - IC: order ning Hidden Class'da "customer" ni qidiradi
    - Monomorphic → IC hit (tez)

  call_site_2: (customer).address
    - IC: customer ning Hidden Class'da "address" ni qidiradi
    - Turli customer shape'lar → polymorphic/megamorphic

  call_site_3: (address).city
    - IC: address ning Hidden Class'da "city" ni qidiradi
    - Yana alohida IC entry
```

"Train wreck" pattern (`a.b.c.d.e.f`) bir nechta muammolar keltiradi:

1. **Har `.` alohida IC cache entry** — V8 ning IC tizimi har call site uchun alohida state saqlaydi. 5 darajali chain = 5 ta alohida IC tracking.
2. **Cascading megamorphism** — agar chain'ning biror darajasida turli Hidden Class'lar ko'rilsa (masalan, turli `customer` tiplari), shu darajadagi IC megamorphic'ga o'tadi, keyingi daraja esa ikkinchi darajali polymorphism'ga duch keladi.
3. **Inline optimization cheklangan** — V8 odatda chain'ni bir butun sifatida emas, har qadamni alohida optimize qiladi. Butun chain'ni inline qilish uchun har bir daraja shape'i barqaror bo'lishi kerak.
4. **Null/undefined safety** — chain'ning istalgan qadamida `undefined` kelsa `TypeError` — optional chaining (`?.`) bu muammoni yumshatadi, lekin design muammosi qolaveradi.

**Delegation orqali yechim** (`customer.getCity()`) — bir call site'da, method inside ichki logic (shu paytda bitta object bilan ishlaydi) — IC uchun **monomorphic va optimize-friendly**. LoD nafaqat "clean code" uchun, balki **engine optimization**'ga ham yordam beradi.

</details>

<summary><strong>Kod Misollari</strong></summary>

## Tell Don't Ask Principle

### Nazariya

**Tell Don't Ask** — ob'ektdan ma'lumot **so'ramang**, keyin uning o'rniga qaror qabul qilmang. Buning o'rniga ob'ektga **nima qilish kerakligini ayting** — qarorni u o'zi qilsin.

Bu prinsip procedural va OOP yondashuvni ajratib turadi:
- **Ask (procedural):** Ma'lumotni ol → tashqarida qaror qil → natijani qayta yoz
- **Tell (OOP):** Ob'ektga vazifani topshir → u o'zi qilsin

<details>
<summary><strong>Under the Hood</strong></summary>

"Tell" paradigmasi V8 JIT optimization uchun **ideal** pattern — u monomorphic method dispatch'ni yaratadi. "Ask" pattern esa bir nechta optimization muammolarni keltirib chiqaradi:

```
Ask pattern:
  if (account.getBalance() >= amount) {
    account.setBalance(account.getBalance() - amount);
  }

V8 nuqtai nazaridan:
  1. getBalance() — 2 marta chaqiruv (duplicated call site work)
  2. getBalance() + setBalance() — 2 ta alohida call site (2 ta IC)
  3. Conditional branch — JIT guard va branch prediction kerak
  4. Logic method tashqarisida → V8 inline qila olmaydi
  5. Hidden class o'zgarish xavfi — har setBalance() shape transition

Tell pattern:
  account.withdraw(amount);

V8 nuqtai nazaridan:
  1. Bitta call site (bitta IC entry)
  2. Method ichida logic — V8 inline qilishi mumkin
  3. Monomorphic IC — optimization ideal
  4. Guards kam — behavior predictable
  5. Hidden class stable — method internal'da shape o'zgarmaydi
```

**V8 inlining:** V8 kichik va barqaror method'larni **inline** qilishi mumkin — ya'ni method'ning bytecode'ini call site'ga to'g'ridan-to'g'ri joylab, call overhead'ni olib tashlaydi. Inline optimizatsiya:

- **Monomorphic call site'larda** avtomatik qo'llaniladi
- **Kichik method'lar** (bytecode kamroq) uchun ehtimoli yuqori
- **Predictable shape'li** receiver'lar uchun samarali

"Tell" style method'lari odatda aniq va kichik bo'lgani uchun V8 JIT ularning ko'pini inline qiladi. "Ask" style code esa kattaroq va tashqi branching ko'proq — inline ehtimoli kam.

**Encapsulation optimizatsiyasi:** "Tell" pattern ob'ektning Hidden Class shape'ini tashqi koddan himoyalaydi. "Ask" pattern'da tashqi kod `account.balance = x` kabi property yozishlar orqali instance shape'ini o'zgartirishi — va **hidden class transition** yaratishi mumkin, bu esa deoptimization va IC invalidation'ga sabab bo'ladi.

</details>

## Composition vs Inheritance

### Nazariya

Ikki yondashuv bor:
- **Inheritance** — "X **is a** Y" (It YO'q hayvon, Hayvon is a Tirik mavjudot)
- **Composition** — "X **has a** Y" (Mashina HAS motori, Shaxs HAS manzili)

Ko'pchilik holatda **composition** afzalroq — chunki u yanada flexible va sust bog'langan (loosely coupled) dizayn beradi.

> **Qoida:** "Favor composition over inheritance" — GoF Design Patterns

```
Inheritance (is-a):              Composition (has-a):

  Animal                           ┌───────────┐
    ├── Dog                        │   Car      │
    │   ├── GuideDog               │  ┌───────┐ │
    │   └── PoliceDog              │  │Engine │ │
    └── Cat                        │  └───────┘ │
        ├── HouseCat               │  ┌───────┐ │
        └── WildCat                │  │ GPS   │ │
                                   │  └───────┘ │
  Chuqur hierarxiya =              └───────────┘
  qattiq bog'liqlik                Flexible, almashtirish oson
```

<details>
<summary><strong>Under the Hood</strong></summary>

Composition va inheritance V8 ning **Hidden Class** tizimiga turlicha ta'sir ko'rsatadi. (V8 source'da Hidden Class `Map` deb nomlanadi — JavaScript `Map`'dan butunlay boshqa tushuncha.) Inheritance da barcha instance'lar bir xil prototype chain'ga ega — shu sababli bir xil Hidden Class'ni share qiladi. Composition da har bir ob'ekt turli xil property'lar kombinatsiyasiga ega bo'lishi mumkin.

```
Inheritance — barcha Dog instance'lar BIR hidden class:

  new Dog("Rex")  → Map0 { name: @0 } → Dog.prototype → Animal.prototype
  new Dog("Buddy") → Map0 { name: @0 } → (aynan shu chain)
  // V8: monomorphic IC, tez property access

Composition — factory har safar yangi shape yaratishi MUMKIN:

  createDog("Rex")   → { eat: fn1, sleep: fn2, bark: fn3, fetch: fn4 }
  createDog("Buddy") → { eat: fn5, sleep: fn6, bark: fn7, fetch: fn8 }
  // Agar closure har safar yangi fn yaratsa → turli hidden class
  // V8: megamorphic IC, sekinroq property access
```

`Object.assign({}, canEat(state), canFly(state))` da har bir spread yangi function object yaratadi (closure). Bu har bir instance uchun alohida hidden class paydo bo'lishiga olib kelishi mumkin. Inheritance da esa method'lar prototype'da bir marta saqlanadi — barcha instance'lar share qiladi.

Lekin V8 composition pattern'ni ham optimizatsiya qila oladi: agar factory function har doim **bir xil tartibda** bir xil property'lar yaratsa, engine transition chain orqali hidden class'ni share qiladi. Property qo'shilish **tartibi** muhim — `{ eat, sleep, bark }` va `{ bark, eat, sleep }` turli hidden class yaratadi.

</details>

<summary><strong>Kod Misollari</strong></summary>

```javascript
// ✅ Composition bilan — kerakli xatti-harakatlarni alohida yig'ish:
const canEat = (state) => ({
  eat: () => `${state.name} yeyapti`
});

const canSleep = (state) => ({
  sleep: () => `${state.name} uxlayapti`
});

const canBark = (state) => ({
  bark: () => `${state.name}: Hav!`
});

const canFetch = (state) => ({
  fetch: (item) => `${state.name} ${item}ni olib kelyapti`
});

// Dog = eat + sleep + bark + fetch

    ...canSleep(state),
```

## OOP vs Functional Programming

### Nazariya

| | **OOP** | **Functional Programming (FP)** |
|---|---|---|
| **Asosiy birlik** | Object (state + behavior) | Function (pure, no side effects) |
| **State** | Mutable (o'zgaruvchan) | Immutable (o'zgarmas) |
| **Ma'lumot oqimi** | Method'lar state'ni o'zgartiradi | Ma'lumot funksiyalardan oqib o'tadi |
| **Qayta ishlatish** | Inheritance, composition | Higher-order functions, composition |
| **Side effects** | Normal (method state o'zgartiradi) | Oldini oladi (pure functions) |
| **Mos keladi** | Domain modeling, UI, game dev | Data transformation, async, math |

JavaScript **ikkala paradigmani** qo'llab-quvvatlaydi — va amalda ko'pincha **aralash** ishlatiladi:

```javascript
// OOP yondashuv:
class Cart {
  #items = [];

  add(item) {
    this.#items.push(item); // state mutate
  }

  get total() {
    return this.#items.reduce((sum, i) => sum + i.price, 0);
  }
}

// FP yondashuv:
const addItem = (cart, item) => [...cart, item]; // yangi array — immutable

const total = (cart) => cart.reduce((sum, i) => sum + i.price, 0); // pure function

// Amalda — aralash (React misol):
// Component = class yoki function (FP)
// State management = immutable updates (FP)
// Service layer = class instances (OOP)
```

## Proxy Bilan OOP — Metaprogramming

### Nazariya

`Proxy` — ob'ektning asosiy operatsiyalarini (property kirish, yozish, o'chirish) **tutib olish** va custom logika qo'shish imkonini beradi. Bu OOP da **encapsulation**, **validation**, va **lazy initialization** ni yangi darajaga olib chiqadi.

> **Batafsil:** Proxy va Reflect haqida to'liq → [23-proxy-reflect.md](23-proxy-reflect.md)

<details>
<summary><strong>Under the Hood</strong></summary>

`Proxy` ECMAScript spec da **exotic object** hisoblanadi — oddiy ob'ektlardan farqli, uning internal method'lari (`[[Get]]`, `[[Set]]`, `[[Has]]` va boshqalar) custom logic bilan override qilingan. `new Proxy(target, handler)` chaqirilganda engine **ProxyCreate** abstract operation ni bajaradi:

```
new Proxy(target, handler):

  1. ProxyCreate(target, handler) chaqiriladi
  2. Yangi exotic object yaratiladi:
     - [[ProxyTarget]] = target
     - [[ProxyHandler]] = handler
  3. Har bir internal method override qilinadi:
     proxy.name  →  [[Get]]:
       a. handler.get trap bormi? → handler.get(target, "name", proxy)
       b. yo'q → target.[[Get]]("name") (default behavior)

     proxy.name = x  →  [[Set]]:
       a. handler.set trap bormi? → handler.set(target, "name", x, proxy)
       b. yo'q → target.[[Set]]("name", x)
```

Proxy **trap invocation** da spec invariant'lar mavjud. Masalan, agar target property `configurable: false, writable: false` bo'lsa, `get` trap boshqa qiymat qaytara olmaydi — engine `TypeError` tashaydi. Bu invariant'lar Proxy ning spec-level xavfsizligini ta'minlaydi.

V8 da Proxy ob'ektlar oddiy ob'ektlardan sekinroq — har bir property access da trap function chaqiruvi kerak. Inline cache Proxy bilan ishlamaydi (trap har safar dynamic). Shu sababli hot path'larda (ko'p marta chaqiriladigan joy) Proxy o'rniga `#` private fields yoki closure ishlatish tezroq.

</details>

<details>
<summary><strong>Kod Misollari</strong></summary>

**Validation — property yozishni nazorat qilish:**

```javascript
function createValidated(target, schema) {
  return new Proxy(target, {
    set(obj, prop, value) {
      const rule = schema[prop];
      if (rule) {
        if (rule.type && typeof value !== rule.type) {
          throw new TypeError(`${prop} ${rule.type} bo'lishi kerak, ${typeof value} berildi`);
        }
        if (rule.min !== undefined && value < rule.min) {
          throw new RangeError(`${prop} minimal: ${rule.min}`);
        }
        if (rule.max !== undefined && value > rule.max) {
          throw new RangeError(`${prop} maksimal: ${rule.max}`);
        }
      }
      obj[prop] = value;
      return true;
    }
  });
}

const user = createValidated({}, {
  name: { type: "string" },
  age:  { type: "number", min: 0, max: 150 },
  email: { type: "string" }
});

user.name = "Ali";       // ✅
user.age = 25;           // ✅
// user.age = -5;        // ❌ RangeError
// user.age = "yigirma"; // ❌ TypeError
```

**Private property — naming convention enforcement:**

```javascript
function createPrivate(target) {
  return new Proxy(target, {
    get(obj, prop) {
      if (typeof prop === "string" && prop.startsWith("_")) {
        throw new Error(`${prop} — private property, kirish taqiq`);
      }
      return obj[prop];
    },
    set(obj, prop, value) {
      if (typeof prop === "string" && prop.startsWith("_")) {
        throw new Error(`${prop} — private, o'zgartirish taqiq`);
      }
      obj[prop] = value;
      return true;
    }
  });
}

const config = createPrivate({
  host: "localhost",
  port: 3000,
  _secretKey: "abc123"

console.log(config.host);       // "localhost" ✅
// console.log(config._secretKey); // ❌ Error: private
```

**Observable pattern — property o'zgarganda notify:**

```javascript
function createObservable(target, onChange) {
  return new Proxy(target, {
    set(obj, prop, value) {
      const oldValue = obj[prop];
      obj[prop] = value;
      if (oldValue !== value) {
        onChange(prop, oldValue, value);
      }
      return true;
    }
  });
}

const state = createObservable({ count: 0, name: "Ali" }, (prop, oldVal, newVal) => {
  console.log(`${prop}: ${oldVal} → ${newVal}`);
});

state.count = 1;    // "count: 0 → 1"
state.name = "Vali"; // "name: Ali → Vali"
state.count = 1;    // (hech narsa — o'zgarmagan)
```

## Edge Cases va Gotchas

### `instanceof` null va primitive'larda — `false` qaytaradi, xato emas

Ko'p dasturchilar `null instanceof Object` yoki `42 instanceof Number` `TypeError` keltiradi deb o'ylaydi. Aslida — **`false` qaytaradi**. `instanceof` faqat RHS (o'ng tomon) non-function bo'lganda `TypeError` tashlaydi.

```javascript
// Primitive va null — doim false:
console.log(null instanceof Object);      // false
console.log(undefined instanceof Object); // false
console.log(42 instanceof Number);         // false — primitive
console.log("text" instanceof String);     // false — primitive
console.log(true instanceof Boolean);      // false — primitive

// Wrapper object'lar — true:
console.log(new Number(42) instanceof Number);    // true
console.log(new String("text") instanceof String); // true

// TypeError faqat RHS non-function bo'lsa:
try {
  const x = {};
  console.log(x instanceof 42); // ❌ TypeError: Right-hand side not callable
} catch (e) {
  console.log(e.message);
}

// Xavfsiz object check:
function isObject(val) {
  return val !== null && typeof val === "object";
}
```

**Yechim:** Object ekanligini tekshirish uchun `instanceof` o'rniga `typeof` va `!== null` kombinatsiyasi ishlating. Primitive'larni class hierarchy bilan aralashtirib yubormaslik.

---

### `Object.freeze()` shallow — nested object'larni freeze qilmaydi

`Object.freeze(obj)` faqat **top-level** property'larni immutable qiladi. Nested object'lar, array'lar, subclass'lar — barchasi hali ham o'zgartirilishi mumkin. Bu "shallow freeze" deb ataladi.

```javascript
class Config {
  constructor() {
    this.host = "localhost";
    this.settings = { level: 1, debug: true };
    this.ports = [3000, 8080];
  }
}

const frozen = Object.freeze(new Config());

// ❌ Top-level — frozen:
frozen.host = "example.com"; // silent fail (strict: TypeError)
console.log(frozen.host); // "localhost"

// ✅ Nested object — HALI HAM o'zgartiriladi:
frozen.settings.level = 99;
console.log(frozen.settings.level); // 99 ❌ — freeze yordam bermadi!

frozen.ports.push(9999);
console.log(frozen.ports); // [3000, 8080, 9999] ❌

// Chuqur freeze uchun o'zingiz yozing:
function deepFreeze(obj) {
  Object.freeze(obj);
  for (const key of Object.keys(obj)) {
    if (typeof obj[key] === "object" && obj[key] !== null) {
      deepFreeze(obj[key]);
    }
  }
  return obj;
}

const deepFrozen = deepFreeze(new Config());
deepFrozen.settings.level = 99; // silent fail
console.log(deepFrozen.settings.level); // 1 ✅
```

**Yechim:** `Object.freeze` shallow ekanligini yodda tuting. Immutability kerak bo'lsa — `deepFreeze` utility funksiyasi, yoki Immutable.js / Immer kabi library ishlating.

---

### Mixin factory har chaqiruvda yangi class yaratadi — identity check muammosi

Subclass factory pattern (`(Base) => class extends Base {}`) har chaqirilganda **yangi** class yaratadi. Ikki bir xil mixin application `===` orqali solishtirganda `false` qaytaradi — bu `instanceof` check'ni ham murakkablashtiradi.

```javascript
const Serializable = (Base) => class extends Base {
  serialize() { return JSON.stringify(this); }
};

const Base = class BaseClass {};

// Har chaqiruv YANGI class:
const Class1 = Serializable(Base);
const Class2 = Serializable(Base);

console.log(Class1 === Class2); // false ❌ — turli class'lar
console.log(Class1.prototype === Class2.prototype); // false

// Ikki alohida class, lekin "bir xil" mixin application:
class UserA extends Serializable(Base) {}
class UserB extends Serializable(Base) {}

const a = new UserA();
const b = new UserB();

console.log(a instanceof UserA); // true
console.log(b instanceof UserA); // false ❌ — turli chain
// Ikkalasida ham serialize() bor, lekin instanceof tekshiruvi alohida
```

**Yechim:** Mixin factory'ni **bir marta** chaqirib, natijani o'zgaruvchiga saqlang:

```javascript
// ✅ Bitta class yarating va ishlating:
const SerializableBase = Serializable(Base);

class UserA extends SerializableBase {}
class UserB extends SerializableBase {}

const a = new UserA();
const b = new UserB();
console.log(a instanceof SerializableBase); // true ✅
console.log(b instanceof SerializableBase); // true ✅
```

Yoki `Symbol.hasInstance` bilan structural check ishlatish (duck typing).

### LSP buzilishi — "narrowing preconditions" klassik misol (Square vs Rectangle)

Liskov Substitution Principle'ning eng mashhur misoli — Square `extends` Rectangle da child class parent contract'ini **toraytiradi**. `Rectangle.setWidth(w)` `width` ni o'zgartirish imkonini beradi. Square o'zining "tomonlari teng" invariantini saqlash uchun `setWidth` da `height` ni ham o'zgartirishga majbur — bu parent contract'ini buzadi.

```javascript
class Rectangle {
  constructor(width, height) {
    this.width = width;
    this.height = height;
  }
  setWidth(w) { this.width = w; }
  setHeight(h) { this.height = h; }
  area() { return this.width * this.height; }
}

// ❌ Square — LSP buzadi:
class Square extends Rectangle {
  constructor(side) { super(side, side); }

  setWidth(w) {
    this.width = w;
    this.height = w; // ❌ Square'ning cheklovi — parent contract'ini o'zgartiradi
  }

  setHeight(h) {
    this.width = h;
    this.height = h;
  }
}

// Rectangle o'rnida Square ishlatilganda — bug:
function resizeAndVerify(rect) {
  rect.setWidth(5);
  rect.setHeight(10);
  // Mantiq bo'yicha 5 * 10 = 50 kutilyapti
  return rect.area();
}

console.log(resizeAndVerify(new Rectangle(1, 1))); // 50 ✅
console.log(resizeAndVerify(new Square(1)));       // 100 ❌ — not 50!
// Square Rectangle o'rnida ishlaytilganida contract buzildi
```

**Yechim:** Square Rectangle'dan `extends` qilmasligi kerak. Ular **boshqa** shape'lar (matematik jihatdan square rectangle bo'lishi mumkin, lekin **software contract** nuqtai nazaridan farqli behavior'lar). To'g'ri yondashuv — ularni alohida Shape subclass qilish yoki composition ishlatish.

**Muhim xulosa:** OOP da **"is-a"** munosabat **matematik** emas, **behavioral**. Agar child parent'ning har qanday method'ini parent kutganidek bajara olmasa — inheritance noto'g'ri tanlov.

### "Tell Don't Ask" query scenarios uchun kuchga kirmaydi

Tell Don't Ask printsipi asosan **command/modification** operatsiyalari uchun qo'llaniladi. Read/query scenarios uchun "ask" qilish **normal va zarur** — ma'lumot olish bu ob'ektdan so'rashdir. Bu printsipni blind holda qo'llash **paranoid encapsulation**'ga olib keladi.

```javascript
class Order {
  #status = "pending";
  #total;
  #items = [];

  constructor(total) { this.#total = total; }

  // ✅ Tell — commands (modification):
  place() { this.#status = "placed"; }
  cancel() { this.#status = "cancelled"; }
  addItem(item) { this.#items.push(item); }

  // ✅ Ask — queries (read) OK va KERAK:
  get status() { return this.#status; }
  get total() { return this.#total; }
  get isEmpty() { return this.#items.length === 0; }
}

// Query scenarios — "Ask" normal:
const orders = [/* ... */];

// ✅ Filter (read operation) — ask kerak:
const pending = orders.filter(o => o.status === "pending");
const large = orders.filter(o => o.total > 1_000_000);
const empty = orders.filter(o => o.isEmpty);

// Filter uchun har order'ga "o'zingni filter qil" deb aytish mumkin emas —
// bu global view, external knowledge talab qiladi
```

**Nima uchun:** CQRS (Command Query Responsibility Segregation) pattern ikkala holatni ajratadi: **commands** (state o'zgartiradi, qiymat qaytarmaydi) va **queries** (o'qiydi, state o'zgartmaydi). Tell Don't Ask **faqat commands** uchun strict qoida. Query'lar uchun esa:

- Getter'lar (`get status`) normal — ichki state'ga controlled access beradi
- Computed property'lar (`get isEmpty`) — ob'ekt o'z ma'lumotini "soddalashtirilgan" ko'rinishda beradi
- Filter/map operations — bu external logic, ob'ekt o'zi filter qilmaydi

**Qoida:** Tell Don't Ask — **"so'ragin-so'rama; keyin qaror qilma"** pattern'ini oldini olish uchun. Oddiy read operatsiyalar istisno.

## Common Mistakes

### 1. Haddan tashqari chuqur inheritance (Deep Hierarchy)

```javascript
// ❌ 5+ darajali inheritance — tushunish va debug qilish qiyin:
class Entity extends Base { }
class LivingEntity extends Entity { }
class Animal extends LivingEntity { }
class Mammal extends Animal { }
class Dog extends Mammal { }
class Labrador extends Dog { }
// Muammo: bitta o'zgartirish butun zanjirni buzishi mumkin

// ✅ 2-3 daraja yetarli, keyin composition ishlatish
```

### 2. Inheritance o'rniga composition kerak bo'lgan joyda inheritance ishlatish

```javascript
// ❌ Stack is-a Array emasligini tushunmaslik:
class Stack extends Array {
  peek() { return this[this.length - 1]; }
}
const stack = new Stack();
stack.push(1);
stack.splice(0, 1); // ❌ Stack da splice bo'lmasligi kerak!

// ✅ Stack HAS-A array (composition):
class Stack {
  #items = [];
  push(item) { this.#items.push(item); }
  pop() { return this.#items.pop(); }
  peek() { return this.#items.at(-1); }
  get size() { return this.#items.length; }
}
```

### 3. God Object — bitta class hamma narsani qiladi

```javascript
// ❌ God Object:
class App {
  login() { /* auth */ }
  fetchData() { /* API */ }
  renderUI() { /* DOM */ }
  sendEmail() { /* email */ }
  generatePDF() { /* PDF */ }
  connectDB() { /* database */ }
}

// ✅ Har bir vazifani alohida class ga ajratish (SRP)
```

### 4. Circular dependency — class'lar bir-biriga bevosita bog'lanishi

```javascript
// ❌ A va B bir-biriga bevosita reference saqlaydi:
class A {
  constructor() {
    this.b = new B(this); // B A ni oladi
  }
  doWork() {
    return this.b.handle(); // A → B
  }
}

class B {
  constructor(a) {
    this.a = a; // ❌ circular — B A ni ushlaydi
  }
  handle() {
    return this.a.doWork(); // ❌ B → A → B → ... infinite recursion
  }
}
// Muammolar: memory leak, stack overflow, test qilish imkonsiz, debug qiyin
```

### ✅ To'g'ri usul: Event Bus orqali decoupling

```javascript
class EventBus {
  #listeners = new Map();
  on(event, fn) {
    if (!this.#listeners.has(event)) this.#listeners.set(event, []);
    this.#listeners.get(event).push(fn);
  }
  emit(event, data) {
    this.#listeners.get(event)?.forEach(fn => fn(data));
  }
}

class A {
  constructor(bus) {
    this.bus = bus;
    bus.on("b-done", (data) => this.handleB(data));
  }
  handleB(data) { console.log("A handled B:", data); }
}

class B {
  constructor(bus) { this.bus = bus; }
  doSomething() {
    this.bus.emit("b-done", { result: 42 }); // A B ni bilmaydi
  }
}

// A va B o'rtasida BEVOSITA reference yo'q — faqat event bus orqali
const bus = new EventBus();
const a = new A(bus);
const b = new B(bus);
b.doSomething(); // "A handled B: { result: 42 }"
```

**Nima uchun:** Circular dependency — OOP'dagi eng yomon pattern'lardan biri. Stack overflow, memory leak (GC reference cycle'ni tozalash qiyin), va tight coupling'ning eng o'tkir ko'rinishi. Yechim: **event-driven** yoki **mediator** pattern — modullar o'rtasida abstraction qatlami.

### 5. Premature abstraction — YAGNI prinsipini buzish

```javascript
// ❌ Hali bitta implementatsiya bor, lekin 4 qatlam abstraction:
class AbstractBaseRepository {
  save(entity) { throw new Error("implement"); }
}

class AbstractDomainRepository extends AbstractBaseRepository {
  findByCriteria(criteria) { throw new Error("implement"); }
}

class AbstractUserRepository extends AbstractDomainRepository {
  findByEmail(email) { throw new Error("implement"); }
}

class UserRepository extends AbstractUserRepository {
  save(user) { /* ... */ }
  findByCriteria(criteria) { /* ... */ }
  findByEmail(email) { /* ... */ }
}

// Muammo: butun hierarxiya faqat bitta implementatsiyani qo'llab-quvvatlash uchun
// Keyinchalik boshqa repository kerak bo'lganda bu struktura mos kelmasligi mumkin
```

### ✅ To'g'ri usul: oddiy boshlang, keyinroq refactor qiling

```javascript
// ✅ Avvalo — minimal, aniq class:
class UserRepository {
  async save(user) { /* ... */ }
  async findByEmail(email) { /* ... */ }
  async findById(id) { /* ... */ }
}

// Kelajakda BookRepository kerak bo'lsa — shu paytda umumiy qism topilsa,
// aynan o'sha paytda abstract class ajratiladi (REAL muammo asosida):

class BookRepository {
  async save(book) { /* ... */ }
  async findByISBN(isbn) { /* ... */ }
}

// Agar ikkalasida ham `save()` va `findById()` umumiy bo'lsa —
// shundagina common base class yoki interface ajratish mantiqli
```

**Nima uchun:** **YAGNI — "You Aren't Gonna Need It"**. SOLID va Design Patterns **real muammolar** uchun, hypothetical future needs uchun emas. Premature abstraction qilish:

- Kodni **tushunish va navigatsiya** qilishni qiyinlashtiradi
- **Noto'g'ri abstraction** yaratishga olib keladi (chunki haqiqiy use case hali yo'q)
- **Maintenance overhead** qo'shadi — abstraction qatlamlarini ham saqlash kerak
- Keyinchalik o'zgartirish **qiyinroq** bo'ladi, chunki abstraction noto'g'ri joyda

**Qoida:** "Rule of Three" — ikkinchi bor duplicatsiya ko'rganda ham abstraction qilma, **uchinchi marta** ko'rganda qil. Abstraction **duplication'dan ham yomonroq** bo'lishi mumkin noto'g'ri tanlansa (Sandi Metz: "Duplication is far cheaper than the wrong abstraction").

## Amaliy Mashqlar

### Mashq 1: Encapsulation Tekshiruv (Oson)

**Savol:** Quyidagi kodning output'ini ayting. `#balance` ga tashqaridan murojaat qilganda nima bo'ladi?

```javascript
class BankAccount {
  #balance;

  constructor(initialBalance) {
    this.#balance = initialBalance;
  }

  get balance() {
    return this.#balance;
  }

  withdraw(amount) {
    if (amount > this.#balance) return false;
    this.#balance -= amount;
    return true;
  }
}

const acc = new BankAccount(1000);
console.log(acc.balance);
console.log(acc.withdraw(300));
console.log(acc.balance);
console.log(acc.withdraw(800));
console.log(acc.balance);

// acc.#balance = 999999; // SyntaxError — parse vaqtida xato!
// Private field'ga class tashqarisidan murojaat qilib bo'lmaydi
```

<details>
<summary>Javob</summary>

```
Output:
1000
true
700
false
700
```

**Eslatma:** Agar kodga `acc.#balance = 999999;` qo'shilsa, bu **parse-time SyntaxError** bo'ladi — butun script parse bo'lmaydi va **hech qanday** console.log ishlamaydi. `try/catch` ham parse-time xatoni ushlay olmaydi. Shu sababli private field'ga tashqi murojaat alohida fayl yoki alohida script ichida sinash mumkin.

```javascript
// acc.balance → getter orqali #balance ni o'qiydi → 1000
// acc.withdraw(300) → 300 <= 1000, #balance = 700, return true
// acc.balance → 700
// acc.withdraw(800) → 800 > 700, return false (o'zgarmaydi)
// acc.balance → 700
```

**Tushuntirish:** `#` bilan e'lon qilingan private field'lar faqat class body ichidan accessible. Tashqi koddan murojaat qilish — **parse-time SyntaxError** beradi. Bu syntax error `try/catch` bilan ushlash mumkin emas, chunki kod bajarilmasdan OLDIN (parse vaqtida) ushlanadi. Bu encapsulation — ichki state ni tashqi koddan himoyalash.
</details>

### Mashq 2: Polymorphism va Method Overriding (O'rta)

**Savol:** Quyidagi kodning output'ini ayting. `describe()` har xil class'larda qanday ishlaydi?

```javascript
class Shape {
  area() {
    throw new Error("area() must be implemented");
  }

  describe() {
    return `${this.constructor.name}: area = ${this.area().toFixed(2)}`;
  }
}

class Circle extends Shape {
  #radius;
  constructor(radius) {
    super();
    this.#radius = radius;
  }
  area() {
    return Math.PI * this.#radius ** 2;
  }
}

class Rectangle extends Shape {
  #width;
  #height;
  constructor(width, height) {
    super();
    this.#width = width;
    this.#height = height;
  }
  area() {
    return this.#width * this.#height;
  }
}

const shapes = [new Circle(5), new Rectangle(4, 6)];

for (const shape of shapes) {
  console.log(shape.describe());
}

try {
  const s = new Shape();
  s.describe();
} catch (e) {
  console.log(e.message);
}
```

### Mashq 3: Composition vs Inheritance (O'rta)

**Savol:** Quyidagi inheritance-based yondashuvda nima muammo bor? Uni composition bilan qayta yozing.

```javascript
class Animal {
  eat() { return "eating"; }
}

class FlyingAnimal extends Animal {
  fly() { return "flying"; }
}

class SwimmingAnimal extends Animal {
  swim() { return "swimming"; }
}

// ❌ Muammo: FlyingFish ham ucha oladi, ham suza oladi
// Lekin JavaScript da multiple inheritance yo'q
// class FlyingFish extends FlyingAnimal, SwimmingAnimal {} — XATO
```

<details>
<summary>Javob</summary>

```javascript
// ✅ Composition bilan yechim — behavior'larni factory function sifatida ajratish:

const canEat = (state) => ({
  eat() { return `${state.name} is eating`; }
});

const canFly = (state) => ({
  fly() { return `${state.name} is flying`; }
});

const canSwim = (state) => ({
  swim() { return `${state.name} is swimming`; }
});

// Factory function'lar — kerakli behavior'larni birlashtiradi:

function createDuck(name) {
  const state = { name };
  return Object.assign(
    {},
    canEat(state),
    canFly(state),
    canSwim(state)
  );
}

function createFish(name) {
  const state = { name };
  return Object.assign(
    {},
    canEat(state),
    canSwim(state)
  );
}

function createFlyingFish(name) {
  const state = { name };
  return Object.assign(
    {},
    canEat(state),
    canSwim(state),
    canFly(state)
  );
}

const duck = createDuck("Duck");
console.log(duck.eat());   // "Duck is eating"
console.log(duck.fly());   // "Duck is flying"
console.log(duck.swim());  // "Duck is swimming"

const flyingFish = createFlyingFish("FlyingFish");
console.log(flyingFish.swim());  // "FlyingFish is swimming"
console.log(flyingFish.fly());   // "FlyingFish is flying"
```

**Tushuntirish:** Inheritance "is-a" munosabatni ifodalaydi — `FlyingFish extends FlyingAnimal` degani "flying fish **bir** flying animal". Lekin flying fish ham ucha oladi, ham suza oladi — bu diamond problem hosil qiladi. JavaScript da multiple inheritance yo'q. Composition "has-a" munosabatni ishlatadi — `flyingFish` ob'ekti `canSwim` va `canFly` **behavior'lariga ega**. Bu yondashuv moslashuvchan: yangi hayvon turi qo'shish uchun faqat kerakli behavior'larni birlashtirish yetarli, inheritance hierarchy'ni o'zgartirishga hojat yo'q.
</details>

### Mashq 4: SOLID — Plugin System (Qiyin)

**Savol:** Quyidagi kod OCP (Open/Closed) va DIP (Dependency Inversion) prinsiplariga mos kelmaydi. Nima uchun? Uni SOLID ga moslab qayta yozing.

```javascript
// ❌ SOLID ga mos kelmaydi:
class NotificationService {
  send(type, message) {
    if (type === "email") {
      console.log(`Email: ${message}`);
    } else if (type === "sms") {
      console.log(`SMS: ${message}`);
    } else if (type === "push") {
      console.log(`Push: ${message}`);
    }
    // Yangi tip qo'shish uchun shu kodni o'zgartirish kerak — OCP buzilgan
    // NotificationService aniq implementatsiyalarga bog'liq — DIP buzilgan
  }
}
```

<details>
<summary>Javob</summary>

```javascript
// ✅ SOLID ga mos yechim:

// Abstraktsiya — har bir notifier shu interface'ga amal qiladi:
class Notifier {
  send(message) {
    throw new Error("send() must be implemented");
  }
}

// Concrete implementatsiyalar — har biri o'z ishini qiladi:
class EmailNotifier extends Notifier {
  send(message) {
    console.log(`Email: ${message}`);
  }
}

class SmsNotifier extends Notifier {
  send(message) {
    console.log(`SMS: ${message}`);
  }
}

class PushNotifier extends Notifier {
  send(message) {
    console.log(`Push: ${message}`);
  }
}

// NotificationService — abstraktsiyaga bog'liq, aniq class'ga emas:
class NotificationService {
  #notifiers = [];

  register(notifier) {
    // ✅ DIP: Notifier abstraktsiyasiga bog'liq
    if (!(notifier instanceof Notifier)) {
      throw new TypeError("Must be a Notifier instance");
    }
    this.#notifiers.push(notifier);
  }

  sendAll(message) {
    // ✅ OCP: yangi notifier qo'shish uchun bu kodni o'zgartirmaydi
    for (const notifier of this.#notifiers) {
      notifier.send(message);
    }
  }
}

// Ishlatish:
const service = new NotificationService();
service.register(new EmailNotifier());
service.register(new SmsNotifier());
service.register(new PushNotifier());

service.sendAll("Server down!");
// Email: Server down!
// SMS: Server down!
// Push: Server down!

// ✅ Yangi tip qo'shish — mavjud kodni O'ZGARTIRMASDAN:
class TelegramNotifier extends Notifier {
  send(message) {
    console.log(`Telegram: ${message}`);
  }
}

service.register(new TelegramNotifier());
service.sendAll("Back online!");
// Email: Back online!
// SMS: Back online!
// Push: Back online!
// Telegram: Back online!
```

**Tushuntirish:** Dastlabki kodda **OCP buzilgan** — yangi notification turi qo'shish uchun `send()` metodidagi `if/else` zanjirini o'zgartirish kerak edi. **DIP buzilgan** — `NotificationService` to'g'ridan-to'g'ri "email", "sms", "push" string'lariga bog'langan, abstraktsiya yo'q. Tuzatilgan versiyada: `Notifier` abstraktsiya sifatida ishlaydi, `NotificationService` faqat `Notifier` interface'iga bog'liq (DIP), yangi tip qo'shish uchun faqat yangi class yozish yetarli — mavjud kod o'zgarmaydi (OCP).
</details>

### Mashq 5: Symbol.hasInstance bilan Interface System (Qiyin)

**Savol:** `defineInterface(methods)` funksiyasini yozing — u **duck-typing based** interface yaratadi. Yaratilgan "interface" object bilan `instanceof` orqali har qanday object tekshirilishi mumkin. Interface barcha talab qilingan method'lar object'da bo'lsa — `true`, aks holda `false` qaytarishi kerak.

```javascript
// Expected API:
const Drawable = defineInterface(["draw", "getBounds"]);
const Printable = defineInterface(["print"]);
const Comparable = defineInterface(["compareTo"]);

class Shape {
  draw() { return "drawing"; }
  getBounds() { return { x: 0, y: 0, w: 100, h: 100 }; }
  print() { return "printing"; }
}

const shape = new Shape();
console.log(shape instanceof Drawable);  // true
console.log(shape instanceof Printable); // true
console.log(shape instanceof Comparable); // false
```

<details>
<summary>Javob</summary>

```javascript
function defineInterface(methods) {
  // Object with Symbol.hasInstance — instanceof ishlatilganda
  // tekshiruvni shu method bajaradi
  return {
    [Symbol.hasInstance](instance) {
      // null/undefined uchun darhol false
      if (instance == null) return false;

      // Har bir method mavjudligini va function ekanligini tekshiruvi
      return methods.every(method =>
        typeof instance[method] === "function"
      );
    },
    // Debug uchun foydali — interface'ning "nomi"
    toString() {
      return `Interface<${methods.join(", ")}>`;
    }
  };
}

// Test:
const Drawable = defineInterface(["draw", "getBounds"]);
const Printable = defineInterface(["print"]);
const Comparable = defineInterface(["compareTo"]);

class Shape {
  draw() { return "drawing"; }
  getBounds() { return { x: 0, y: 0, w: 100, h: 100 }; }
  print() { return "printing"; }
}

const shape = new Shape();
console.log(shape instanceof Drawable);   // true
console.log(shape instanceof Printable);  // true
console.log(shape instanceof Comparable); // false — compareTo yo'q

// Plain object bilan ham ishlaydi (duck typing):
const button = {
  draw() { return "button drawn"; },
  getBounds() { return { x: 10, y: 10, w: 80, h: 30 }; },
  print() { return "button printed"; }
};

console.log(button instanceof Drawable);  // true ✅
console.log(button instanceof Printable); // true ✅

// null/undefined/primitive check:
console.log(null instanceof Drawable);    // false ✅
console.log(42 instanceof Drawable);      // false ✅
console.log("text" instanceof Drawable);  // false ✅

// Generic funksiya — har qanday Drawable bilan ishlaydi:
function renderAll(items) {
  return items
    .filter(item => item instanceof Drawable)
    .map(item => item.draw());
}

console.log(renderAll([shape, button, { name: "Ali" }]));
// ["drawing", "button drawn"] — Ali object Drawable emas
```

**Tushuntirish:** `Symbol.hasInstance` well-known symbol `instanceof` operator ichki mexanizmini override qiladi. `x instanceof Drawable` yozilganda engine avval `Drawable[Symbol.hasInstance]` ni qidiradi — topsa shu method chaqiriladi, default prototype chain check o'rniga. Bu pattern **structural typing** (duck typing) ni `instanceof` syntax'i bilan birlashtiradi — class hierarchy shart emas, method mavjudligi kifoya.

Bu yondashuv TypeScript interface'lariga o'xshaydi, lekin **runtime'da** ishlaydi. Real-world use case'lar: plugin system'lar, dependency injection container, duck-typed polymorphism (hechqanday class hierarchy talab qilmaydi).
</details>
