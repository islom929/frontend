## Fetch API Chuqur

### Nazariya

Fetch API — HTTP so'rovlar yuborish uchun zamonaviy, Promise-based interfeys. XMLHttpRequest o'rniga kelgan. **Muhim**: fetch faqat tarmoq xatosida reject bo'ladi — 404/500 kabi HTTP xatolar reject **bo'lmaydi**, `response.ok` ni doim tekshiring.

### Asosiy Ishlatish

```javascript
// GET
const response = await fetch("https://api.example.com/users");
const data = await response.json();

// POST
const response = await fetch("https://api.example.com/users", {
  method: "POST",
  headers: { "Content-Type": "application/json" },
  body: JSON.stringify({ name: "Ali", age: 25 })
});
```

### Headers, Request, Response

```javascript
// === Headers ===
const headers = new Headers();
headers.append("Content-Type", "application/json");
headers.append("Authorization", "Bearer token123");
headers.has("Content-Type"); // true
for (const [key, value] of headers) console.log(`${key}: ${value}`);

// === Request ===
const request = new Request("https://api.example.com/data", {
  method: "POST",
  headers: new Headers({ "Content-Type": "application/json" }),
  body: JSON.stringify({ key: "value" }),
  mode: "cors",          // cors | no-cors | same-origin
  credentials: "include", // include | same-origin | omit
  cache: "no-cache",     // default | no-cache | reload | force-cache
  redirect: "follow"     // follow | error | manual
});
const response = await fetch(request);

// === Response ===
console.log(response.ok);         // true (200-299)
console.log(response.status);     // 200
console.log(response.statusText); // "OK"
console.log(response.redirected); // redirect bo'lganmi?

// Body — BIR MARTA o'qiladi!
const json = await response.json();
// YOKI: .text(), .blob(), .arrayBuffer(), .formData()
```

### AbortController — So'rovni Bekor Qilish

```javascript
const controller = new AbortController();

const fetchPromise = fetch("/api/large-data", {
  signal: controller.signal
});

setTimeout(() => controller.abort(), 5000); // 5s timeout

try {
  const response = await fetchPromise;
  const data = await response.json();
} catch (error) {
  if (error.name === "AbortError") {
    console.log("So'rov bekor qilindi");
  }
}

// Timeout helper
function fetchWithTimeout(url, options = {}, timeout = 5000) {
  const controller = new AbortController();
  const id = setTimeout(() => controller.abort(), timeout);
  return fetch(url, { ...options, signal: controller.signal })
    .finally(() => clearTimeout(id));
}
```

### Error Handling — Network vs HTTP Errors

```javascript
async function safeFetch(url) {
  try {
    const response = await fetch(url);
    if (!response.ok) {
      throw new Error(`HTTP ${response.status}: ${response.statusText}`);
    }
    return await response.json();
  } catch (error) {
    if (error.name === "AbortError") {
      console.log("Bekor qilindi");
    } else if (error.name === "TypeError") {
      console.error("Tarmoq xatosi:", error); // server yoqilmagan, CORS
    } else {
      console.error("HTTP xatosi:", error.message);
    }
    throw error;
  }
}
```

### Streaming Responses

```javascript
async function downloadWithProgress(url) {
  const response = await fetch(url);
  const total = parseInt(response.headers.get("Content-Length"), 10);
  const reader = response.body.getReader();

  let received = 0;
  const chunks = [];

  while (true) {
    const { done, value } = await reader.read();
    if (done) break;
    chunks.push(value);
    received += value.length;
    console.log(`${total ? Math.round(received / total * 100) : "?"}%`);
  }

  return new Blob(chunks);
}
```

## FormData

### Nazariya

`FormData` — form ma'lumotlarini yig'ish va yuborish uchun API. `multipart/form-data` formatida ishlaydi — fayl yuklash uchun ham ishlatiladi. Fetch bilan yuborishda `Content-Type` header'ni **qo'ymang** — brauzer `boundary` bilan birga o'zi qo'yadi.

<details>
<summary><strong>Under the Hood</strong></summary>

FormData `fetch()` ga body sifatida berilganda, brauzer uni `multipart/form-data` formatiga encode qiladi. Har bir field alohida "part" sifatida joylashadi, ular orasida **boundary** string ajratib turadi. Boundary — brauzer generate qiladigan random string, masalan `------WebKitFormBoundaryABC123`. Bu boundary `Content-Type` header ichiga qo'yiladi: `Content-Type: multipart/form-data; boundary=----WebKitFormBoundaryABC123`.

```
Wire format:
------WebKitFormBoundaryABC123\r\n
Content-Disposition: form-data; name="username"\r\n
\r\n
Ali\r\n
------WebKitFormBoundaryABC123\r\n
Content-Disposition: form-data; name="avatar"; filename="photo.jpg"\r\n
Content-Type: image/jpeg\r\n
\r\n
<binary data>\r\n
------WebKitFormBoundaryABC123--
```

Agar `Content-Type` ni qo'lda qo'ysangiz, boundary yo'qoladi — server body ni parse qila olmaydi. Shuning uchun brauzerga o'zi qo'yishiga ruxsat berish kerak. FormData internal'da `name → value[]` map sifatida saqlanadi. `append()` qo'shadi, `set()` esa o'sha key'ning barcha qiymatlarini o'chirib bitta yangi qiymat qo'yadi. File qiymatlari `Blob` subclass sifatida saqlanadi — `name`, `lastModified`, `type` metadata bilan birga.

</details>

<details>
<summary><strong>Kod Misollari</strong></summary>

```javascript
// HTML form dan
const formData = new FormData(document.getElementById("myForm"));

// Qo'lda yaratish
const data = new FormData();
data.append("name", "Ali");
data.append("hobbies", "dasturlash");
data.append("hobbies", "kitob"); // bir key da ko'p qiymat

// Fayl qo'shish
const fileInput = document.getElementById("avatar");
data.append("avatar", fileInput.files[0]);

// Fetch — Content-Type avtomatik!
await fetch("/api/register", {
  method: "POST",
  body: data // headers qo'ymang!
});

// Usullar
data.get("name");        // "Ali"
data.getAll("hobbies");  // ["dasturlash", "kitob"]
data.has("age");         // false
data.set("name", "Vali"); // almashtiradi
data.delete("hobbies");

// Object ga
const obj = Object.fromEntries(data);
// Ko'p qiymatli key'lar yo'qoladi — faqat oxirgisi
```

</details>

## Blob va File API

### Nazariya

`Blob` (Binary Large Object) — xom binary data ni saqlash uchun immutable object. `File` — Blob'ning kengaytmasi, fayl nomi va o'zgartirish vaqtini qo'shadi. `FileReader` — fayllarni asinxron o'qish API'si. Rasmlarni preview qilish, fayllarni yuklab olish, va binary data bilan ishlash uchun zarur.

### Blob

```javascript
// Blob yaratish
const blob = new Blob(["Salom, dunyo!"], { type: "text/plain" });
console.log(blob.size); // 13
console.log(blob.type); // "text/plain"

// Blob dan URL yaratish
const url = URL.createObjectURL(blob);
// "blob:http://localhost/abc-123-def"

// img.src = url; yoki a.href = url qilib ishlatish mumkin

// URL ni tozalash (memory leak oldini olish)
URL.revokeObjectURL(url);

// Blob dan matn o'qish
const text = await blob.text();

// Blob ni slice qilish
const chunk = blob.slice(0, 5, "text/plain"); // birinchi 5 byte
```

### File va FileReader

```javascript
// <input type="file" id="fileInput" multiple>
const input = document.getElementById("fileInput");
input.addEventListener("change", (e) => {
  const files = e.target.files; // FileList

  for (const file of files) {
    console.log(file.name);         // "photo.jpg"
    console.log(file.size);         // 1024000
    console.log(file.type);         // "image/jpeg"
    console.log(file.lastModified); // timestamp
  }
});

// FileReader — faylni o'qish
function readFile(file) {
  return new Promise((resolve, reject) => {
    const reader = new FileReader();
    reader.onload = () => resolve(reader.result);
    reader.onerror = () => reject(reader.error);
    reader.readAsDataURL(file); // base64 string
    // Yoki: readAsText(), readAsArrayBuffer()
  });
}

// Rasm preview
const dataUrl = await readFile(imageFile);
document.getElementById("preview").src = dataUrl;

// Drag and Drop bilan fayl olish
dropzone.addEventListener("drop", (e) => {
  e.preventDefault();
  const files = e.dataTransfer.files;
  handleFiles(files);
});
dropzone.addEventListener("dragover", (e) => e.preventDefault());
```

```javascript
function downloadBlob(blob, filename) {
```

## IntersectionObserver

### Nazariya

`IntersectionObserver` — element viewport ga ko'rinib qolganini kuzatadi. Scroll event ishlatmasdan, performance samarali.

<details>
<summary><strong>Under the Hood</strong></summary>

IntersectionObserver brauzerning rendering pipeline'ida **compositor thread** darajasida ishlaydi. Scroll event'dan farqi — har bir scroll frame'da main thread'da callback chaqirilmaydi. Brauzer o'zi element'larning bounding rect'larini kuzatib, faqat threshold'lar kesishganda callback'ni **asynchronous** ravishda chaqiradi.

```
Rendering Pipeline:
  Style → Layout → Paint → Composite
                              ↑
                    IntersectionObserver shu yerda ishlaydi
                    (layout hisoblangandan keyin, lekin
                     main thread'ni bloklamasdan)
```

**Threshold calculation:** `threshold: [0, 0.25, 0.5, 0.75, 1]` berilsa, brauzer target element'ning `boundingClientRect` va root'ning `rootBounds` orasidagi **intersection ratio** ni hisoblaydi: `intersectionArea / targetArea`. Ratio threshold qiymatlardan birini kesib o'tganda callback fire bo'ladi.

**Root margin** CSS margin sintaksisida beriladi (`"100px 0px"`) va root bounding box'ni virtually kengaytiradi — element hali viewport'ga kirmay turib callback olish imkonini beradi (preloading uchun).

Internal'da brauzer `IntersectionObserverRegistration` ro'yxatini yuritadi. Har bir rendering frame'da (odatda `requestAnimationFrame` vaqtida) barcha registered target'larning intersection'lari batch hisoblanadi. Callback'lar microtask emas, **task queue** ga qo'yiladi — shuning uchun `setTimeout` bilan bir xil priority'da ishlaydi.

</details>

<details>
<summary><strong>Kod Misollari</strong></summary>

```javascript
const observer = new IntersectionObserver((entries) => {
  entries.forEach(entry => {
    if (entry.isIntersecting) {
      console.log("Ko'rindi:", entry.target);
      observer.unobserve(entry.target);
    }
  });
}, {
  root: null,        // null = viewport
  rootMargin: "0px", // preload uchun margin
  threshold: 0       // 0 = biror qismi ko'ringanda
});

observer.observe(document.querySelector(".target"));
```

</details>

### Lazy Loading Images

```javascript
function lazyLoadImages() {
  const images = document.querySelectorAll("img[data-src]");

  const observer = new IntersectionObserver((entries) => {
    entries.forEach(entry => {
      if (entry.isIntersecting) {
        const img = entry.target;
        img.src = img.dataset.src;
        img.removeAttribute("data-src");
        observer.unobserve(img);
      }
    });
  }, { rootMargin: "200px" });

  images.forEach(img => observer.observe(img));
}
```

### Infinite Scroll

```javascript
function setupInfiniteScroll() {
  const sentinel = document.getElementById("scroll-sentinel");
  let page = 1;
  let loading = false;

  const observer = new IntersectionObserver(async (entries) => {
    if (entries[0].isIntersecting && !loading) {
      loading = true;
      page++;
      const data = await fetch(`/api/posts?page=${page}`).then(r => r.json());
      if (data.posts.length === 0) {
        observer.unobserve(sentinel);
        return;
      }
      const fragment = document.createDocumentFragment();
      data.posts.forEach(post => {
        const div = document.createElement("div");
        div.textContent = post.title;
        fragment.appendChild(div);
      });
      sentinel.before(fragment);
      loading = false;
    }
  });

  observer.observe(sentinel);
}
```

## MutationObserver

### Nazariya

`MutationObserver` — DOM o'zgarishlarini asinxron kuzatadi: element qo'shildi/o'chirildi (`childList`), attribute o'zgardi (`attributes`), matn o'zgardi (`characterData`).

<details>
<summary><strong>Under the Hood</strong></summary>

MutationObserver callback'lari **microtask queue** da bajariladi — Promise `.then()` bilan bir xil priority. DOM mutation sodir bo'lganda, brauzer `MutationRecord` objectlarni internal **record queue** ga yig'adi. Hozirgi JavaScript execution context tugagandan keyin (call stack bo'shaganda), microtask checkpoint'da barcha yig'ilgan record'lar bitta callback chaqiruvida batch sifatida beriladi.

```
JS execution:
  div.appendChild(child1)  → record queue: [MutationRecord1]
  div.setAttribute("class") → record queue: [MutationRecord1, MutationRecord2]
  div.appendChild(child2)  → record queue: [MutationRecord1, MutationRecord2, MutationRecord3]
  // ... call stack bo'shadi ...
  // microtask: observer.callback([MR1, MR2, MR3]) — BITTA chaqiruv
```

Bu dizayn eski `Mutation Events` (DOM3) dan tubdan farq qiladi — u har bir mutation uchun sinxron event fire qilardi va cascading mutation'lar infinite loop yaratishi mumkin edi. MutationObserver esa mutation'lar tugaguncha kutadi, so'ng batch callback beradi.

Har bir `MutationRecord` ichida: `type` (childList/attributes/characterData), `target` (o'zgargan node), `addedNodes`/`removedNodes` (NodeList), `previousSibling`/`nextSibling`, `attributeName`, `oldValue` (agar `attributeOldValue: true` / `characterDataOldValue: true` berilgan bo'lsa). `subtree: true` recursive kuzatish yoqadi — observer DOM subtree'dagi barcha descendant node'larga ham registration qo'shadi.

`observer.takeRecords()` — hali callback'ga berilmagan pending record'larni sinxron olish va queue'ni tozalash. `disconnect()` — observer'ni to'xtatadi va pending record'larni ham tashlab yuboradi.

</details>

<details>
<summary><strong>Kod Misollari</strong></summary>

```javascript
const observer = new MutationObserver((mutations) => {
  mutations.forEach(mutation => {
    switch (mutation.type) {
      case "childList":
        mutation.addedNodes.forEach(node => console.log("Qo'shildi:", node));
        mutation.removedNodes.forEach(node => console.log("O'chirildi:", node));
        break;
      case "attributes":
        console.log(`${mutation.attributeName} o'zgardi`);
        break;
      case "characterData":
        console.log("Matn o'zgardi:", mutation.target.data);
        break;
    }
  });
});

observer.observe(document.getElementById("app"), {
  childList: true,
  attributes: true,
  characterData: true,
  subtree: true,              // recursive
  attributeFilter: ["class"],  // faqat shu attribute'lar
  attributeOldValue: true
});

observer.disconnect(); // to'xtatish
```

</details>

## ResizeObserver

```javascript
const observer = new ResizeObserver((entries) => {
```

## PerformanceObserver

### Nazariya

`PerformanceObserver` — performance entry'larni real-time kuzatadi. Navigation timing, resource loading, user timing (marks/measures), long tasks va Core Web Vitals o'lchash uchun ishlatiladi.

<details>
<summary><strong>Under the Hood</strong></summary>

Brauzer ichida **Performance Timeline** degan global buffer mavjud. Har bir sahifa yuklash, resurs download, user timing mark/measure va boshqa performance event'lar `PerformanceEntry` subclass'lari sifatida bu timeline'ga yoziladi. PerformanceObserver shu buffer'ga observer pattern orqali ulanadi.

```
Performance Timeline Buffer:
┌────────────────────────────────────────────────────────┐
│ PerformanceNavigationTiming (sahifa yuklash)           │
│ PerformanceResourceTiming  (script.js, style.css, ...) │
│ PerformanceMark            (user-defined marks)        │
│ PerformanceMeasure         (user-defined measures)     │
│ PerformanceLongTaskTiming  (50ms+ tasklar)             │
│ LargestContentfulPaint     (LCP entry)                 │
│ LayoutShift                (CLS entry)                 │
│ PerformanceEventTiming     (INP entry)                 │
└────────────────────────────────────────────────────────┘
         ↑                              ↑
    buffered: true                 real-time
  (oldingi entry'lar ham)      (yangi entry'lar)
```

`performance.now()` **High Resolution Time** spec'iga asoslangan — `DOMHighResTimeStamp` qaytaradi (microsecond aniqlikda double). `Date.now()` dan farqi: monotonic (system clock o'zgarishiga bog'liq emas) va `timeOrigin` (sahifa navigation boshlanishi) ga nisbatan hisoblanadi.

`buffered: true` opsiyasi muhim — observer yaratilishidan oldin yozilgan entry'larni ham olish imkonini beradi. Busiz faqat observer yaratilgandan keyingi entry'lar keladi. Resource timing buffer default 250 ta entry saqlaydi — to'lganda `resourcetimingbufferfull` event fire bo'ladi, `performance.clearResourceTimings()` bilan tozalash kerak.

Observer callback'lari **task queue** da bajariladi (microtask emas). Entry'lar brauzerning internal instrumentation'idan keladi — rendering engine, network stack va task scheduler'dan.

</details>

<details>
<summary><strong>Kod Misollari</strong></summary>

```javascript
// Long Task kuzatish (50ms+ tasklar)
const longTaskObserver = new PerformanceObserver((list) => {
  list.getEntries().forEach(entry => {
    console.warn(`Long task: ${entry.duration.toFixed(0)}ms`);
  });
});
longTaskObserver.observe({ type: "longtask", buffered: true });

// Resource Loading kuzatish
const resourceObserver = new PerformanceObserver((list) => {
  list.getEntries().forEach(entry => {
    console.log(`${entry.name}: ${entry.duration.toFixed(0)}ms`);
  });
});
resourceObserver.observe({ type: "resource", buffered: true });

// User Timing
performance.mark("start-render");
// ... render ...
performance.mark("end-render");
performance.measure("render-time", "start-render", "end-render");

const timingObserver = new PerformanceObserver((list) => {
  list.getEntries().forEach(entry => {
    console.log(`${entry.name}: ${entry.duration.toFixed(2)}ms`);
  });
});
timingObserver.observe({ type: "measure" });

// performance.now() — yuqori aniqlikdagi vaqt
const t0 = performance.now();
someExpensiveOperation();
console.log(`${(performance.now() - t0).toFixed(2)}ms`);
```

</details>

## Web Workers

### Nazariya

Web Workers — JavaScript ni **alohida thread** da ishlatish. Main thread (UI) bloklanmaydi. Worker'da DOM ga kirish **mumkin emas**. Ma'lumot almashinuvi `postMessage()` orqali (Structured Clone algoritmi).

### Turlari

| Tur | Ishlatish | DOM? |
|-----|-----------|------|
| **Dedicated Worker** | Bitta sahifa, og'ir hisoblar | Yo'q |
| **Shared Worker** | Bir necha tab orasida | Yo'q |
| **Service Worker** | Offline cache, push | Yo'q |

### Dedicated Worker

```javascript
// === main.js ===
const worker = new Worker("worker.js");

worker.postMessage({ type: "calculate", data: [1, 2, 3, 4, 5] });

worker.onmessage = (e) => {
  console.log("Natija:", e.data.result);
};

worker.onerror = (error) => {
  console.error("Worker xatosi:", error.message);
};

// worker.terminate(); // to'xtatish
```

```javascript
// === worker.js ===
self.onmessage = (e) => {
  const { type, data } = e.data;
  if (type === "calculate") {
    const result = data.reduce((sum, n) => sum + n, 0);
    self.postMessage({ result });
  }
};
```

### Shared Worker — Bir Nechta Tab Orasida

**Shared Worker** — bir xil origin'dagi **bir nechta browsing context'lar** (tab'lar, iframe'lar, window'lar) orasida birgalikda ishlatiladigan worker. Dedicated Worker'dan farqli — har bir tab o'z worker instance'ini yaratmaydi, **bitta worker instance**'i barcha ulanuvchi tab'lar bilan ishlaydi. Communication `MessagePort` orqali — har bir ulanish (tab) uchun alohida port.

**Use cases:**
- Bitta WebSocket connection'ni bir nechta tab orasida share qilish (multi-tab chat, real-time dashboard)
- Cross-tab state sinxronlashuvi (login state, cart, preferences)
- Centralized caching — bir marta fetch qilib, barcha tab'larga tarqatish
- Background processing bir nechta tab'dan ma'lumotlar bilan

```javascript
// === main.js (har bir tab'da ishlaydi) ===
const worker = new SharedWorker("shared-worker.js", { name: "my-shared" });

// Port orqali communication
worker.port.start(); // muhim — explicit start

worker.port.postMessage({ type: "hello", tabId: Date.now() });

worker.port.onmessage = (e) => {
  console.log("Shared worker dan:", e.data);
};

// Har bir tab worker.port orqali o'zining portini oladi,
// lekin server-side bitta SharedWorker instance
```

```javascript
// === shared-worker.js ===
const connections = []; // barcha ulanuvchi tab'lar ro'yxati
let sharedState = { count: 0, users: [] };

// Har yangi tab ulanganda
self.onconnect = (e) => {
  const port = e.ports[0]; // shu tab'ning porti
  connections.push(port);

  port.onmessage = (event) => {
    const { type, tabId } = event.data;

    if (type === "hello") {
      sharedState.users.push(tabId);
      // Barcha tab'larga yangi holat broadcast qilish
      connections.forEach(p => p.postMessage({
        type: "state-update",
        state: sharedState
      }));
    }

    if (type === "increment") {
      sharedState.count++;
      // Barcha tab'larga yangilangan count
      connections.forEach(p => p.postMessage({
        type: "count-update",
        count: sharedState.count
      }));
    }
  };

  port.start();
};
```

**Dedicated vs Shared Worker farqi:**

| Xususiyat | Dedicated Worker | Shared Worker |
|-----------|------------------|---------------|
| Instance | Har tab alohida | Bitta instance, ko'p tab |
| Communication | `worker.onmessage` direct | `worker.port` orqali |
| Lifecycle | Tab yopilsa tugaydi | Oxirgi tab yopilsa tugaydi |
| `onconnect` | Yo'q | Har yangi tab ulanganda trigger |
| DevTools | Tab DevTools'da | Alohida `chrome://inspect` ichida |
| Browser support | Keng | Safari kam qo'llab-quvvatlaydi |

**Eslatma:** Shared Worker'lar Safari'da **to'liq qo'llab-quvvatlanmaydi** (2024 holati bo'yicha). Chrome, Firefox, Edge'da to'liq ishlaydi. Cross-browser compatibility kerak bo'lsa, `BroadcastChannel API` + `Service Worker` kombinatsiyasi muqobil yechim.

## WebSocket API

### Nazariya

WebSocket — brauzer va server orasida **to'liq duplex** (ikki tomonlama) persistent ulanish. HTTP so'rovlardan farqli — ulanish bir marta ochiladi va ikki tomon istalgan vaqtda data yuborishi mumkin. Real-time ilovalar uchun: chat, o'yinlar, live data, collaborative editing.

<details>
<summary><strong>Under the Hood</strong></summary>

WebSocket ulanishi HTTP **Upgrade handshake** bilan boshlanadi. Brauzer oddiy HTTP request yuboradi, lekin maxsus header'lar bilan:

```
Client → Server:
GET /socket HTTP/1.1
Host: example.com
Upgrade: websocket
Connection: Upgrade
Sec-WebSocket-Key: dGhlIHNhbXBsZSBub25jZQ==   (random base64)
Sec-WebSocket-Version: 13

Server → Client:
HTTP/1.1 101 Switching Protocols
Upgrade: websocket
Connection: Upgrade
Sec-WebSocket-Accept: s3pPLMBiTxaQ9kYGzzhZRbK+xOo=  (SHA-1 hash)
```

`Sec-WebSocket-Accept` server tomonidan `Sec-WebSocket-Key + "258EAFA5-E914-47DA-95CA-C5AB0DC85B11"` concatenation'ining SHA-1 hash'i sifatida hisoblanadi. Bu CSRF himoyasi emas — handshake validation uchun.

101 response'dan keyin TCP connection HTTP dan WebSocket **frame protocol** ga o'tadi. Har bir message frame ichida: opcode (0x1 = text, 0x2 = binary, 0x8 = close, 0x9 = ping, 0xA = pong), payload length (7-bit, 16-bit yoki 64-bit), masking key (client → server yo'nalishda majburiy, XOR mask), va payload data.

Brauzer internal'da bitta TCP socket'ni ikki yo'nalishda ishlatadi — alohida send va receive buffer'lar bilan. `ws.send()` chaqirilganda data brauzerning network thread'idagi **send buffer** ga qo'yiladi va asinxron yuboriladi. `ws.bufferedAmount` property hali yuborilmagan baytlar sonini ko'rsatadi — flow control uchun foydali.

`readyState` davlat mashinasi: `CONNECTING(0)` → `OPEN(1)` → `CLOSING(2)` → `CLOSED(3)`. Close handshake ikki tomonlama — bir tomon close frame yuboradi, ikkinchi tomon close frame bilan javob beradi, keyin TCP connection yopiladi.

</details>

<details>
<summary><strong>Kod Misollari</strong></summary>

```javascript

// ws.readyState: 0=CONNECTING, 1=OPEN, 2=CLOSING, 3=CLOSED

  console.log("Ulandi");
  ws.send("Salom server!");
  ws.send(JSON.stringify({ type: "join", room: "general" }));

  console.log("Yopildi:", event.code, event.reason);
  // event.code: 1000=normal, 1001=going away, 1006=abnormal
  // Reconnect logic qo'shish mumkin

```

</details>

### Auto-Reconnect Pattern

```javascript
function createWebSocket(url) {
  let ws;
  let retries = 0;
  const maxRetries = 5;

  function connect() {
    ws = new WebSocket(url);

    ws.addEventListener("open", () => {
      retries = 0; // reset
    });

    ws.addEventListener("close", () => {
      if (retries < maxRetries) {
        const delay = Math.min(1000 * 2 ** retries, 30000); // exponential backoff
        retries++;
        setTimeout(connect, delay);
      }
    });

    ws.addEventListener("message", (e) => {
      console.log("Xabar:", JSON.parse(e.data));
    });
  }

  connect();
  return {
    send: (data) => ws.readyState === 1 && ws.send(JSON.stringify(data)),
    close: () => { retries = maxRetries; ws.close(); }
  };
}
```

## Server-Sent Events (SSE)

### Nazariya

`EventSource` — server dan client ga **bir tomonlama** stream. WebSocket dan soddaroq — HTTP asosida, avtomatik reconnect, faqat text data. Live feed, notifications, progress update uchun ideal.

<details>
<summary><strong>Under the Hood</strong></summary>

SSE oddiy HTTP GET response ustida ishlaydi. Server `Content-Type: text/event-stream` header bilan javob beradi va connection'ni **yopmaydi** — chunked transfer encoding bilan data yuborilib turadi. Brauzer response body'ni stream sifatida o'qiydi va `\n\n` (ikki newline) bilan ajratilgan event'larni parse qiladi.

```
HTTP Response (connection ochiq qoladi):
Content-Type: text/event-stream
Cache-Control: no-cache
Connection: keep-alive

data: {"msg": "salom"}\n\n           ← default "message" event
event: notification\n               ← custom event type
data: {"type": "alert"}\n\n
id: 42\n                             ← event ID (reconnect uchun)
data: ping\n\n
retry: 5000\n\n                      ← reconnect interval (ms)
```

Brauzer `EventSource` internal'da **last event ID** ni saqlaydi. Agar connection uzilsa, brauzer avtomatik reconnect qiladi va `Last-Event-ID` HTTP header'ini yuboradi — server shu ID'dan keyingi event'larni qayta yuborishi mumkin (missed message recovery).

`retry:` field brauzerga reconnect interval'ini o'rgatadi (default ~3 soniya). Reconnect faqat network error yoki server connection yopganda sodir bo'ladi — agar brauzer `source.close()` chaqirsa, reconnect bo'lmaydi. `readyState`: `CONNECTING(0)`, `OPEN(1)`, `CLOSED(2)`.

SSE HTTP/2 multiplexing bilan yaxshi ishlaydi — bitta TCP connection'da ko'plab stream'lar ochish mumkin. HTTP/1.1 da brauzerning per-domain connection limiti (6 ta) muammo bo'lishi mumkin — har bir EventSource bitta connection egallaydi.

</details>

<details>
<summary><strong>Kod Misollari</strong></summary>

```javascript
const source = new EventSource("/api/stream");

source.addEventListener("message", (event) => {
  console.log("Data:", event.data);
});

source.addEventListener("notification", (event) => {
  // Custom event type
  const data = JSON.parse(event.data);
  showNotification(data);
});

source.addEventListener("error", () => {
  if (source.readyState === EventSource.CLOSED) {
    console.log("Stream yopildi");
  }
  // Avtomatik reconnect bo'ladi
});

source.close(); // to'xtatish
```

</details>

| | WebSocket | SSE | Long Polling |
|---|-----------|-----|-------------|
| Yo'nalish | Ikki tomonlama | Server → Client | Request-Response |
| Protocol | ws:// / wss:// | HTTP | HTTP |
| Reconnect | Qo'lda | Avtomatik | Har request |
| Binary data | Ha | Yo'q (faqat text) | Ha |
| Ishlatish | Chat, o'yinlar | Live feed, notifications | Legacy support |

## requestIdleCallback

### Nazariya

`requestIdleCallback(callback, options?)` — brauzerga "quyidagi ishni **bo'sh vaqt** bo'lganda bajar" deyish uchun API. Brauzer har frame'ning oxirida (paint/composite tugagandan keyin) keyingi frame boshlanishigacha qolgan vaqt borligini tekshiradi — agar bo'sh vaqt bo'lsa, `requestIdleCallback` callback'ini chaqiradi. Agar bo'sh vaqt yo'q bo'lsa — keyingi frame'ga qoldiradi.

Asosiy maqsad: **past-priority ishlarni main thread jank'ini keltirib chiqarmasdan bajarish**. Analytics, logging, prefetch, DOM cleanup, cache invalidation, background indexing kabi "muhim lekin shoshilinch emas" operatsiyalar uchun ideal.

**Callback'ga `IdleDeadline` object beriladi** — undan ikki metod/property:
- `deadline.timeRemaining()` — bo'sh vaqt qolgani (ms, odatda 0-50 atrofida)
- `deadline.didTimeout` — `timeout` berilgan va u o'tib ketganmi (true bo'lsa ish qat'iy bajarilishi kerak)

**`timeout` option** muhim — brauzer bo'sh vaqt hech qachon topmasa (uzluksiz band bo'lsa), callback cheksiz kutadi. `timeout: 2000` qo'ysangiz — 2 soniyadan keyin brauzer majburan chaqiradi, `deadline.didTimeout === true` bilan.

<details>
<summary><strong>Under the Hood</strong></summary>

Brauzer rendering cycle: `Event → JavaScript → Style → Layout → Paint → Composite → rAF → ... → IDLE → next frame`. `requestIdleCallback` aynan **IDLE** bo'limida ishlaydi — bu bo'lim uzunligi frame budget'ga bog'liq (60Hz'da ~16.67ms frame, agar JS + style + layout + paint 5ms olgan bo'lsa — 11.67ms bo'sh qoladi).

`requestAnimationFrame` bilan farqi:
- **rAF** — har frame'ning **boshida**, paint'dan oldin ishlaydi → vizual update'lar uchun
- **rIC** — frame oxirida, **bo'sh vaqt bo'lsa** ishlaydi → past-priority ish uchun

Bo'sh vaqt yo'q bo'lsa (brauzer band — ko'p animatsiya, og'ir JS, user input ko'p) — callback deadline yo'q bo'lsa uzoq kutishi mumkin. Shuning uchun **`timeout` doim qo'shish tavsiya etiladi**.

**Browser support:** Chrome, Firefox, Edge — to'liq. Safari **qo'llab-quvvatlamaydi** (2024 holatiga). Safari uchun polyfill: `setTimeout(cb, 1)` bilan fallback (lekin bu native rIC behavior'ni simulate qilmaydi — faqat async execution).

Facebook React'ning "Scheduler" paketi ilgari rIC ishlatgan, lekin cross-browser compatibility va precision'dan noroziligi uchun o'zining MessageChannel-based scheduling'ni yaratgan. Bu ko'rsatadi — production'da rIC'ni to'g'ridan-to'g'ri ishlatish emas, yuqori darajadagi task scheduling pattern ichida foydalanish yaxshiroq.

</details>

<details>
<summary><strong>Kod Misollari</strong></summary>

```javascript
// Production'da Scheduler API yoki MessageChannel-based scheduler afzal
```

## Concurrency — SharedArrayBuffer va Atomics

### Nazariya

**SharedArrayBuffer** — main thread va Web Worker'lar o'rtasida **umumiy xotira** (shared memory) yaratish imkonini beradi. Oddiy `postMessage` bilan ma'lumot yuborilganda **copy** qilinadi — katta data uchun bu sekin. SharedArrayBuffer esa bir xil xotira maydonchasini ikkala thread ham to'g'ridan-to'g'ri o'qiy/yoza oladi — copy yo'q.

**Atomics** — shared memory ustida **thread-safe** operatsiyalar bajarish uchun static metodlar to'plami. Agar ikkita thread bir vaqtda bitta xotira joyiga yozsa — **race condition** yuzaga keladi. Atomics bu muammoni hal qiladi — operatsiyalar **atomic** (bo'linmas) bo'lib bajariladi.

**Asosiy Atomics metodlari:**
- `Atomics.load(arr, index)` — atomic o'qish (boshqa thread yozayotgan paytda ham to'g'ri qiymat olish)
- `Atomics.store(arr, index, value)` — atomic yozish
- `Atomics.add(arr, index, value)` — atomic qo'shish (load + add + store bir operatsiyada)
- `Atomics.sub/and/or/xor` — atomic arifmetik/bitwise operatsiyalar
- `Atomics.wait(arr, index, value)` — `arr[index] === value` bo'lsa `notify()` chaqirilguncha bloklanib kutadi; teng bo'lmasa darhol `"not-equal"` qaytaradi (faqat Worker thread'da — main thread'da **ishlamaydi**, browser'ni freezing'dan himoya qilish uchun)
- `Atomics.notify(arr, index, count)` — kutayotgan thread'larni uyg'otish
- `Atomics.waitAsync(arr, index, value)` (ES2022) — `Atomics.wait` ning non-blocking versiyasi, Promise qaytaradi, main thread'da ham ishlaydi

**Cross-Origin Isolation:** SharedArrayBuffer ishlatish uchun sahifa **cross-origin isolated** bo'lishi kerak. Server quyidagi HTTP header'larni yuborishi shart:
- `Cross-Origin-Opener-Policy: same-origin` (COOP)
- `Cross-Origin-Embedder-Policy: require-corp` (COEP)
- Bu header'lar bo'lmasa `SharedArrayBuffer` undefined bo'ladi

**Use cases:** game engine'larda physics/rendering parallel, real-time audio processing, katta data set'lar ustida parallel hisoblash, image/video processing.

<details>
<summary><strong>Under the Hood</strong></summary>

SharedArrayBuffer oddiy ArrayBuffer'dan farqli ravishda **process-level shared memory** yaratadi. Oddiy `postMessage(buffer)` da Structured Clone algoritmi buffer'ning to'liq copy'sini yaratadi. `postMessage(sharedBuffer)` da esa faqat **reference** yuboriladi — ikkala thread bir xil physical memory region'ga pointer oladi.

```
Oddiy ArrayBuffer:                SharedArrayBuffer:
Main Thread    Worker             Main Thread    Worker
┌─────────┐   ┌─────────┐       ┌─────────┐   ┌─────────┐
│ buffer A │   │ buffer B │       │ ptr ────────────→ ptr │
│ [1,2,3]  │   │ [1,2,3]  │       └────┬────┘   └────┬────┘
└─────────┘   └─────────┘            │              │
  (copy)        (copy)               ▼              ▼
                                ┌──────────────────┐
                                │  Shared Memory   │
                                │  [1, 2, 3]       │
                                └──────────────────┘
```

Atomics metodlari CPU ning **atomic instructions** ga map qilinadi (x86'da `lock` prefix bilan `CMPXCHG`, `XADD` va boshqalar). `Atomics.add(arr, 0, 1)` bitta CPU instruction sifatida bajariladi — boshqa thread o'rtada qiymatni o'zgartira olmaydi. Bu hardware-level guarantee.

`Atomics.wait()` internal'da OS ning **futex** (fast userspace mutex) mehanizmiga asoslangan. Thread haqiqatan ham sleep holatiga o'tadi — CPU cycle sarflamaydi. `Atomics.notify()` OS kernel orqali sleeping thread'ni uyg'otadi. `Atomics.waitAsync()` esa futex o'rniga Promise-based polling ishlatadi — main thread'da ishlaydi, chunki main thread'ni bloklash mumkin emas.

Spectre va Meltdown hujumlari sababli SharedArrayBuffer 2018 da vaqtincha disable qilindi. Cross-Origin Isolation (COOP + COEP header'lar) kiritilgandan keyin qayta yoqildi — bu header'lar sahifani boshqa origin'lar bilan process'ni share qilishdan himoya qiladi.

</details>

## Page Visibility API

### Nazariya

Page Visibility API — foydalanuvchi sahifani ko'rayotganini yoki boshqa tab/ilovaga o'tganini aniqlash imkonini beradi. Bu API orqali resurs'larni tejash, foydalanuvchi tajribasini yaxshilash va aniqroq analytics to'plash mumkin.

**document.visibilityState** — sahifaning hozirgi holati:
- `"visible"` — sahifa ko'rinib turibdi (active tab, minimized emas)
- `"hidden"` — sahifa yashirin (boshqa tab, minimized, lock screen)

**document.hidden** — `true` yoki `false` qaytaradi. `visibilityState === "hidden"` bilan bir xil.

**visibilitychange event** — sahifa holati o'zgarganda fire bo'ladi. Bu event `document` da tinglash kerak (`window` da emas).

**Use cases:**
- Video/audio — tab yashiringanda pause qilish, qaytganda davom ettirish
- Polling/WebSocket — yashirin tab da so'rovlarni to'xtatish yoki sekinlashtirish, resurlarni tejash
- Analytics — foydalanuvchi sahifada qancha vaqt **haqiqatan** o'tkazganini o'lchash
- Animatsiya — `requestAnimationFrame` yashirin tab da avtomatik to'xtaydi, lekin `setInterval` davom etadi — qo'lda to'xtatish kerak
- Notification — tab yashirin bo'lganda yangi xabar kelsa notification ko'rsatish

<details>
<summary><strong>Under the Hood</strong></summary>

Page Visibility API brauzerning **tab management** va **window management** subsystem'lariga bog'langan. `document.visibilityState` qiymati brauzer process'ining OS-level window state'idan olinadi: tab switch, window minimize, lock screen, va boshqa OS event'lar.

```
OS/Browser Event              →  visibilityState
─────────────────────────────────────────────────
Boshqa tab ga o'tish          →  "hidden"
Tab ga qaytish                →  "visible"
Window minimize               →  "hidden"
Window restore                →  "visible"
Lock screen                   →  "hidden"
Sahifa prerender (bfcache)    →  "hidden"
```

`visibilitychange` event `document` object'da fire bo'ladi. Brauzer ichida bu event OS'ning window focus/blur signal'laridan trigger bo'ladi. Event **synchronous** dispatch bo'ladi — `visibilityState` property callback ichida yangi qiymatni ko'rsatadi.

Brauzer yashirin tab'lar uchun **throttling** qo'llaydi. Ko'rinadigan tab'da ham nested (5+ chuqurlik) `setTimeout`/`setInterval` HTML spec bo'yicha **~4ms** ga clamp qilinadi. Yashirin tab'larda Chrome va Firefox odatda timer'larni **~1 sekunda** darajasida sekinlashtiradi, battery saver yoki aggressive tab discarding rejimida yana sekinroq. `requestAnimationFrame` to'liq to'xtatiladi. Audio context suspend bo'ladi. Precision throttling rules browser va versiyaga bog'liq — Chrome'ning "Intensive WakeUp Throttling" (2021+) background tab'larni keskin cheklaydi. Bu optimizatsiyalar CPU va battery tejash uchun.

`document.hidden` deprecated emas, lekin `document.visibilityState` ko'proq ma'lumot beradi (kelajakda yangi state'lar qo'shilishi mumkin). `beforeunload` va `unload` event'lar mobile brauzer'larda ishonchsiz — `visibilitychange` bilan `"hidden"` state tekshirish eng ishonchli usul sahifa yopilishini aniqlash uchun.

</details>

## Broadcast Channel API

### Nazariya

**BroadcastChannel** — bitta origin (domain) dagi barcha tab'lar, iframe'lar va Worker'lar o'rtasida xabar almashish API'si. Oddiy, sodda interfeys — channel yaratib, `postMessage` bilan xabar yuborish va `onmessage` bilan qabul qilish.

**Ishlash prinsipi:**
- `new BroadcastChannel("channel-name")` — kanal yaratish yoki mavjudiga ulanish
- `channel.postMessage(data)` — xabar yuborish (barcha boshqa tab'lardagi SHU NOMDAGI kanal'lar oladi)
- `channel.onmessage = (e) => {}` — xabar qabul qilish
- `channel.close()` — kanaldan chiqish
- **Muhim:** xabar yuborgan tab o'zi eshitmaydi — faqat boshqa tab'lar

**Use cases:**
- **Logout sync** — bitta tab da logout qilsa, barcha tab'lar ham logout bo'ladi
- **Theme sync** — bitta tab da dark mode yoqsa, barchasida o'zgaradi
- **Data sync** — bitta tab da ma'lumot yangilansa, barchasida refresh
- **Cart sync** — e-commerce da savatchani barcha tab'larda sinxronlash

**vs localStorage event:** `window.addEventListener("storage", ...)` ham tab'lar arasi communication qiladi, lekin farqlari:
- localStorage — faqat string saqlash, serialization kerak
- BroadcastChannel — har qanday structured-cloneable data (object, array, Blob va boshqalar)
- localStorage — faqat storage o'zgarishini eshitadi, maxsus xabar tuzilmasi yo'q
- BroadcastChannel — maxsus channel va xabar tuzilmasi bor, semantic aniqroq

<details>
<summary><strong>Under the Hood</strong></summary>

BroadcastChannel brauzerning **browsing context group** darajasida ishlaydi. Bir xil origin'dagi barcha tab'lar, iframe'lar va Worker'lar bitta group ichida. `new BroadcastChannel("name")` chaqirilganda, brauzer internal channel registry'da shu nom bilan yozilgan barcha port'lar ro'yxatini oladi.

```
Tab 1: new BroadcastChannel("auth")  ─┐
Tab 2: new BroadcastChannel("auth")  ─┼─→  Internal Channel Registry
Tab 3: new BroadcastChannel("auth")  ─┘    origin + name = channel group
Worker: new BroadcastChannel("auth") ─┘

Tab 1: postMessage({type: "LOGOUT"})
  → Tab 2: onmessage fires
  → Tab 3: onmessage fires
  → Worker: onmessage fires
  → Tab 1: ESHITMAYDI (o'zi yuborgan)
```

`postMessage()` chaqirilganda data **Structured Clone Algorithm** orqali serialize/deserialize bo'ladi — har bir receiver o'z copy'sini oladi (shared memory emas). Shuning uchun object, array, Blob, ArrayBuffer, Map, Set va boshqa structured-cloneable type'lar yuborish mumkin, lekin Function, DOM node, Error kabi type'lar **clone qilib bo'lmaydi**.

Message delivery **asynchronous** — task queue orqali. Brauzer IPC (Inter-Process Communication) ishlatadi: Chrome da har bir tab alohida process'da ishlaydi, shuning uchun xabarlar OS-level IPC channel (pipe/socket) orqali yuboriladi. Bu `localStorage` event'dan farqi — localStorage'da disk I/O kerak, BroadcastChannel esa faqat in-memory IPC.

`channel.close()` chaqirilganda shu port registry'dan o'chiriladi. Agar GC port'ni yig'ib olsa ham, port avtomatik close bo'ladi — lekin `close()` ni explicit chaqirish best practice.

</details>

<summary><strong>Kod Misollari</strong></summary>

## Beacon API

### Nazariya

**navigator.sendBeacon()** — sahifa yopilayotganda yoki foydalanuvchi boshqa sahifaga o'tayotganda, **ishonchli** tarzda kichik ma'lumot yuborish uchun mo'ljallangan API.

**Nima uchun Fetch/XHR emas?**
- `unload` yoki `beforeunload` event'ida yuborilgan `fetch`/`XHR` so'rovlar browser tomonidan **cancel** qilinishi mumkin — sahifa yopilayotgan bo'lsa, browser so'rovni kutmaydi
- `sendBeacon()` so'rovi browser tomonidan **background'da** yuboriladi — sahifa yopilgandan keyin ham so'rov muvaffaqiyatli yuborilishiga kafolat beradi
- Browser so'rovni o'zi boshqaradi — developer kutish, retry yoki error handling bilan shug'ullanmasligi kerak

**Xususiyatlari:**
- Faqat **POST** metodi
- Kichik payload — odatda 64KB gacha (browser'ga qarab farqlaydi)
- Return qiymati `boolean` — `true` (so'rov queue ga qo'shildi) yoki `false` (xato)
- Body sifatida: string, `Blob`, `FormData`, `URLSearchParams`, `ArrayBuffer` qabul qiladi
- **CORS** — server `Access-Control-Allow-Origin` header'ini qaytarishi kerak (cross-origin holatda)

**Use cases:** analytics event'lar (sahifa yopilish, session tugash), A/B test natijalari, error reporting, foydalanuvchi xatti-harakati tracking.

<details>
<summary><strong>Under the Hood</strong></summary>

`sendBeacon()` ichida brauzer so'rovni **browser process**'ning network queue'siga qo'yadi — renderer process (JavaScript ishlayotgan joy) emas. Shuning uchun sahifa yopilganda ham so'rov yuboriladi: renderer process yo'q qilinadi, lekin browser process'dagi network queue'dan so'rov allaqachon yuborila boshlagan.

```
Oddiy fetch() unload'da:           sendBeacon():
┌──────────────┐                   ┌──────────────┐
│  Renderer     │                   │  Renderer     │
│  Process      │                   │  Process      │
│  fetch() ──X──│── sahifa yopildi  │  sendBeacon() │
│  (cancel)     │                   │    │          │
└──────────────┘                   └────┼─────────┘
                                        ▼
                                   ┌──────────────┐
                                   │  Browser      │
                                   │  Process      │
                                   │  network queue│──→ Server
                                   │  (tirik qoladi)│
                                   └──────────────┘
```

Internal'da `sendBeacon()` HTTP POST request yaratadi. Body `Content-Type` avtomatik belgilanadi: string → `text/plain`, Blob → Blob'ning `type` property'si, FormData → `multipart/form-data`, URLSearchParams → `application/x-www-form-urlencoded`. ArrayBuffer/TypedArray uchun Content-Type qo'yilmaydi.

So'rov `keepalive` flag bilan yuboriladi — bu fetch API'dagi `keepalive: true` option bilan bir xil mehanizm. Brauzer keepalive so'rovlar uchun umumiy **64KB** limit qo'yadi (barcha pending keepalive so'rovlar jami). Bu limitdan oshsa `sendBeacon()` `false` qaytaradi.

CORS qoidalari oddiy fetch bilan bir xil ishlaydi. Agar cross-origin server'ga yuborilsa va `Content-Type` "simple" bo'lmasa (masalan `application/json` Blob orqali), preflight OPTIONS so'rov kerak bo'ladi — bu sahifa yopilish vaqtida muammoli bo'lishi mumkin.

</details>

## Web Vitals va Performance Metrics

### Nazariya

**Core Web Vitals** — Google ning web sahifa sifatini o'lchaydigan asosiy metrikalar:
- **LCP (Largest Contentful Paint)** — sahifadagi eng katta element (rasm, matn blok) qachon render bo'lganini o'lchaydi. Yaxshi: < 2.5s. LCP elementlari: `<img>`, `<video>`, `background-image`, katta matn bloklari
- **INP (Interaction to Next Paint)** — foydalanuvchi interaksiyasi (click, tap, keyboard) dan keyingi visual update'gacha bo'lgan vaqt. FID o'rniga kelgan (2024). Yaxshi: < 200ms. Barcha interaksiyalarning eng yomoni hisoblanadi
- **CLS (Cumulative Layout Shift)** — sahifa yuklanayotganda elementlar qancha "sakrab" joyini o'zgartirishini o'lchaydi. Yaxshi: < 0.1. Sabablari: o'lchamlari ko'rsatilmagan rasmlar, dynamic content, web font'lar, dom manipulyatsiya

**Performance API:**
- `performance.now()` — yuqori aniqlikdagi timestamp (microsecond), `Date.now()` dan aniqroq
- `performance.mark("label")` — vaqt belgisi qo'yish
- `performance.measure("name", "startMark", "endMark")` — ikki belgi orasidagi vaqtni o'lchash
- `performance.getEntriesByType("measure")` — barcha o'lchov natijalarini olish

**PerformanceObserver** — performance entry'larni kuzatish uchun observer pattern. Entry type'lar:
- `"largest-contentful-paint"` — LCP kuzatish
- `"layout-shift"` — CLS kuzatish
- `"event"` — INP kuzatish
- `"navigation"` — sahifa yuklash metrikalar
- `"resource"` — individual resurs (rasm, script, CSS) yuklash vaqtlari
- `"longtask"` — 50ms dan uzun task'lar (jank detection)

**Navigation Timing** — sahifa yuklash bosqichlari: DNS lookup, TCP connect, request/response, DOM processing, load event. `performance.getEntriesByType("navigation")[0]` bilan olish mumkin.

**Resource Timing** — har bir resurs (script, stylesheet, image) uchun alohida yuklash metrikalar. `performance.getEntriesByType("resource")` bilan olish mumkin.

**Long Animation Frames API** — 50ms dan uzun animation frame'larni aniqlash (jank/stutter sababi). `"long-animation-frame"` entry type bilan kuzatish mumkin. LoAF API `longtask` API ning yangilangan versiyasi — qaysi script qaysi funksiya sababchi ekanini ko'rsatadi.

**web-vitals library** — Google ning rasmiy kutubxonasi. Core Web Vitals'ni o'lchash uchun tayyor funksiyalar: `onLCP()`, `onINP()`, `onCLS()`, `onFCP()`, `onTTFB()`. npm: `web-vitals`.

<details>
<summary><strong>Under the Hood</strong></summary>

Core Web Vitals brauzerning rendering pipeline'ining turli bosqichlaridan olinadi:

```
Navigation → DNS → TCP → TLS → Request → Response → Parse → Style → Layout → Paint → Composite
                                  ↑                                     ↑          ↑
                                TTFB                                   LCP     CLS (shift)

User Input → Event Handler → Style → Layout → Paint → Composite → Next Frame
     ↑                                                      ↑
     └──────────── INP (bu oraliq) ─────────────────────────┘
```

**LCP** — brauzer har bir paint operatsiyasida eng katta element'ni track qiladi. `LargestContentfulPaint` entry'lari ketma-ket fire bo'ladi (har safar yangi eng katta element topilganda). Oxirgi entry haqiqiy LCP hisoblanadi. Foydalanuvchi birinchi input yoki scroll qilganda LCP reporting to'xtaydi — interaction bo'ldi degani sahifa "loaded" deb hisoblanadi.

**INP** — brauzer har bir discrete user event (click, keydown, tap) uchun `PerformanceEventTiming` entry yozadi. `duration` = event timestamp'dan keyingi paint'gacha. INP qiymati sahifadagi barcha interaction'larning ichidan **"eng yomon"ga yaqin** tanlanadi: aniq formula `web-vitals` library'sida — har 50 interaction'ga 1 tasini "istisno" qilib, shu formula asosida yuqori percentile (taxminan 98th percentile) qiymati qaytariladi. 50'dan kam interaction bo'lgan sessiyalarda eng yomon duration ishlatiladi. Bu FID'dan farqli — FID faqat birinchi input'ning **input delay**'ini o'lchardi, INP esa barcha interaction'larning to'liq duration'ini (input delay + processing time + presentation delay).

**CLS** — brauzer har bir layout recalculation'dan keyin element'larning oldingi va yangi pozitsiyalarini solishtiradi. `LayoutShift` entry'dagi `value` = `impact fraction * distance fraction`. Impact fraction — viewport'ning qancha qismi ta'sirlangani. Distance fraction — element qancha masofa siljigani. `hadRecentInput` 500ms ichida user input bo'lganini bildiradi — bunday shift'lar CLS'ga qo'shilmaydi (foydalanuvchi o'zi trigger qilgan).

**Navigation Timing** brauzerning network stack'idan olinadi: DNS resolver, TCP socket pool, TLS handshake, HTTP parser — har bir bosqich `PerformanceNavigationTiming` entry'da microsecond aniqlikda qayd qilinadi.

## Edge Cases va Gotchas

Browser API'lar bo'yicha 5 ta nozik, production'da tez-tez uchrab, debug qilish qiyin bo'lgan gotcha.

### Gotcha 1: `fetch` Response body faqat bir marta o'qiladi — `clone()` yechimi

`Response` object body'si **stream** asosida ishlaydi. `response.json()`, `response.text()`, `response.blob()`, `response.arrayBuffer()` — barchasi body stream'ini **consume** qiladi. Bir marta o'qigandan keyin ikkinchi marta chaqirish `TypeError: Body has already been consumed` beradi.

```javascript
const response = await fetch("/api/data");

// ❌ Ikki marta o'qish — xato!
const data = await response.json();
const text = await response.text(); // TypeError: Body already consumed

// ❌ Error handling paytida body'ni qayta o'qish
const response = await fetch("/api/data");
try {
  const data = await response.json();
  return data;
} catch (e) {
  // JSON parse xatosi bo'lsa text sifatida ko'rishni hohlaymiz
  const errorText = await response.text(); // TypeError — allaqachon consumed
  console.error("Raw response:", errorText);
}

// ✅ Yechim 1: response.clone() ikkinchi reader uchun
const response = await fetch("/api/data");
const textCopy = response.clone(); // body'ni clone qilish (o'qishdan oldin!)

try {
  const data = await response.json();
  return data;
} catch (e) {
  const raw = await textCopy.text(); // clone'dan o'qish ✅
  console.error("JSON parse failed:", raw);
}

// ✅ Yechim 2: faqat bir marta o'qish va natijani saqlash
const response = await fetch("/api/data");
const text = await response.text(); // avval text
try {
  const data = JSON.parse(text); // so'ng parse
  return data;
} catch (e) {
  console.error("Invalid JSON:", text); // text allaqachon bor
}
```

**Nima uchun:** Fetch API spec'i body'ni `ReadableStream` sifatida tushunadi — stream faqat bir marta o'qiladi (head'dan tail'gacha). Bu memory efficient — server'dan kelayotgan gigabyte'lab data to'liq yuklashni kutmasdan stream sifatida ishlatish mumkin. `clone()` method shu stream'ni duplicate qiladi, lekin bu body'ni memory'ga fully buffer qilishini talab qiladi — juda katta response'lar uchun samarador emas.

**Yechim:** Har doim `clone()` ni **body o'qishdan oldin** chaqiring. Error handling uchun body'ni text sifatida olib, keyin `JSON.parse()` qilish eng toza pattern.

### Gotcha 2: `localStorage.setItem` — `QuotaExceededError` silent bo'lmaydi

`localStorage.setItem` quota (5-10MB origin'da) yetmasa **`DOMException: QuotaExceededError`** tashlaydi — aytib qo'ymaydi va data saqlanmaydi. Ko'pincha dasturchilar bu xatoni kutmaydi va production'da user data yo'qolishi mumkin.

```javascript
// ❌ Quota xatosini kutmaslik
function saveUserData(data) {
  localStorage.setItem("userData", JSON.stringify(data));
  // Quota yetmasa — silent fail! (aslida exception tashlaydi lekin
  // top-level call bo'lsa dastur crash bo'ladi)
}

// ❌ Try/catch'siz loop
for (const item of manyItems) {
  localStorage.setItem(`item-${item.id}`, JSON.stringify(item));
  // 3000 item'dan keyin quota tugaydi — loop o'rtada to'xtaydi
}

// ✅ Try/catch bilan safe setItem
function safeSetItem(key, value) {
  try {
    localStorage.setItem(key, JSON.stringify(value));
    return true;
  } catch (e) {
    if (e.name === "QuotaExceededError" ||
        e.name === "NS_ERROR_DOM_QUOTA_REACHED") {
      console.warn("localStorage quota yetdi:", key);
      // Eski ma'lumotlarni tozalash, yoki user'ga bildirish
      return false;
    }
    throw e; // boshqa xato bo'lsa qayta tashlash
  }
}

// ✅ Quota tekshirish API (zamonaviy)
async function checkStorageQuota() {
  if (!navigator.storage || !navigator.storage.estimate) {
    return null; // eski browser
  }
  const { usage, quota } = await navigator.storage.estimate();
  console.log(`Ishlatilgan: ${(usage / 1024 / 1024).toFixed(2)} MB`);
  console.log(`Mavjud: ${(quota / 1024 / 1024).toFixed(2)} MB`);
  console.log(`Foiz: ${(usage / quota * 100).toFixed(1)}%`);
  return { usage, quota };
}

// ✅ localStorage cleanup pattern
function ensureSpace() {
  const keys = Object.keys(localStorage);
  // Eski cache item'larni o'chirish
  keys.filter(k => k.startsWith("cache-"))
    .sort() // LRU assume qilish uchun timestamp bilan
    .slice(0, 10)
    .forEach(k => localStorage.removeItem(k));
}
```

**Nima uchun:** Web Storage spec'i `setItem` aniq exception tashlashni talab qiladi — lekin modal yoki notification yo'q. Private/Incognito mode'da quota ~0 ga yaqin bo'lishi mumkin (ba'zi browser'larda). Safari Private mode `setItem`'ni bloklaydi va xato tashlaydi — ko'pchilik saytlar buni handle qilmaydi va crash beradi.

**Yechim:** Doim `localStorage.setItem` chaqiriqlarini try/catch'ga o'rash. Kritik data uchun `navigator.storage.estimate()` bilan quota tekshirish va proactive cleanup pattern ishlatish. Katta data uchun `IndexedDB`'ga ko'chish (unda ham quota bor, lekin boshqarish oson).

### Gotcha 3: `MutationObserver` callback ichida DOM mutatsiyasi — reentrance xavfi

`MutationObserver` callback'i microtask'da ishlaydi va unga batch qilingan mutation'lar keladi. Agar callback ichida kuzatilayotgan DOM'ni mutatsiya qilsangiz — yangi mutation'lar **yangi queue'ga** qo'shiladi va **keyingi callback chaqirig'ida** yetkaziladi (cheksiz emas, lekin cascade). Noto'g'ri yozilgan callback **infinite loop** yaratishi mumkin.

```javascript
const target = document.getElementById("container");

// ❌ Infinite loop xavfi:
const observer = new MutationObserver((mutations) => {
  mutations.forEach(m => {
    if (m.type === "childList") {
      // Har yangi child uchun class qo'shmoqchimiz...
      m.addedNodes.forEach(node => {
        if (node.nodeType === 1) {
          node.classList.add("processed");
          // ⚠️ Bu "attributes" mutation'ni trigger qiladi
          // → yangi MutationRecord → keyingi callback chaqiriq
          // Agar biz "attributes" ni ham kuzatib "classList.remove" qilsak — loop!
        }
      });
    }
  });
});

observer.observe(target, {
  childList: true,
  attributes: true, // attributeFilter siz — har attribute kuzatiladi
  subtree: true
});

// ✅ Yechim 1: attributeFilter bilan o'zimizning mutation'larni e'tiborsiz qoldirish
const observer = new MutationObserver((mutations) => {
  mutations.forEach(m => {
    if (m.type === "childList") {
      m.addedNodes.forEach(node => {
        if (node.nodeType === 1) node.classList.add("processed");
      });
    }
  });
});

observer.observe(target, {
  childList: true,
  attributes: true,
  attributeFilter: ["data-status"], // faqat shu attribute'ni kuzatish
  subtree: true
});

// ✅ Yechim 2: Flag bilan reentrance'ni oldini olish
let processing = false;
const observer = new MutationObserver((mutations) => {
  if (processing) return; // reentrance'ni bloklash
  processing = true;

  try {
    mutations.forEach(m => {
      // Mutatsiyalarni handle qilish
      applyChanges(m);
    });
  } finally {
    // Keyingi microtask'da flag'ni tiklash
    queueMicrotask(() => { processing = false; });
  }
});

// ✅ Yechim 3: takeRecords() bilan pending mutation'larni tozalash
const observer = new MutationObserver((mutations) => {
  // Bizning mutation'larni amalga oshirish
  doOwnMutations();

  // Bizning mutation'lardan kelgan record'larni olib tashlash
  observer.takeRecords(); // pending'larni drain qiladi, callback'ga yuborilmaydi
});
```

**Nima uchun:** MutationObserver'ning batch + microtask semantikasi eski "Mutation Events" (DOM3) ning cascade muammosini hal qilish uchun yaratilgan. Lekin hali ham callback ichida mutation mumkin — ular keyingi microtask'da yetadi. `attributes: true` berganingizda **barcha** attribute o'zgarishlarini kuzatadi, shu jumladan o'zingiz qilgan `classList.add`/`classList.remove` ham — bu cascade manbai.

**Yechim:** Eng muhimi — `attributeFilter` bilan faqat kerak bo'lgan attribute'larni kuzatish. `takeRecords()` bizning mutation'larni "drain" qilib, keyingi callback'ga yubormaydi. Reentrance flag pattern ham ishlaydi, lekin `queueMicrotask` bilan flag'ni tiklash muhim — sync reset cascade'ga olib kelishi mumkin.

### Gotcha 4: `IntersectionObserver.observe()` birinchi callback'ni darhol chaqiradi

Ko'pchilik `observe()` chaqirganda "scroll boshlanganda javob kelishini" kutadi. Aslida brauzer **darhol** (keyingi microtask'da) current intersection state bilan callback'ni chaqiradi — element allaqachon viewport'da bo'lsa ham, tashqarida bo'lsa ham. Bu initial "state snapshot" callback.

```javascript
// ❌ Initial callback'ni kutmaslik xato assumption'larga olib keladi:
const observer = new IntersectionObserver((entries) => {
  console.log("Scroll bo'ldi"); // ← AYNAN IKKI MARTA CHAQIRILADI:
  // 1. observe() dan keyin darhol (initial state)
  // 2. Haqiqiy scroll paytida
  trackScrollEvent(); // ← ikki marta fire bo'ladi!
});

observer.observe(element);

// ❌ Faqat "kirdi" event kutayotgan kod:
const observer2 = new IntersectionObserver((entries) => {
  entries.forEach(entry => {
    if (entry.isIntersecting) {
      showAnimation(); // ← element dastlab viewport'da bo'lsa darhol ishlaydi!
    }
  });
});

// ✅ Yechim 1: Initial state'ni ignore qilish (agar kerak bo'lsa)
let initialized = false;
const observer = new IntersectionObserver((entries) => {
  if (!initialized) {
    initialized = true;
    return; // birinchi callback'ni skip qilish
  }
  entries.forEach(entry => {
    if (entry.isIntersecting) showAnimation();
  });
});

// ✅ Yechim 2: Initial state'ni ham handle qilish (odatda shu kerak)
const observer = new IntersectionObserver((entries) => {
  entries.forEach(entry => {
    if (entry.isIntersecting) {
      // Element ko'rinyapti — animatsiya ko'rsatish
      // (dastlab ko'rinsa ham, scroll bilan kirsa ham — bir xil logika)
      showAnimation(entry.target);
      observer.unobserve(entry.target); // faqat birinchi marta
    }
  });
});

// ✅ Lazy loading uchun — initial callback foydali!
// Dastlab viewport'da bo'lgan rasmlar darhol yuklansin:
document.querySelectorAll("img[data-src]").forEach(img => {
  observer.observe(img); // callback darhol ishlaydi → yuqoridagi rasmlar darhol loaded
});
```

**Nima uchun:** `IntersectionObserver` spec'i "initial observation" deb ataladigan behavior'ni talab qiladi — `observe()` chaqirilgan element uchun brauzer current intersection state'ini callback'ga yuboradi. Bu by design — lazy loading uchun ayni muddao (dastlab ko'rinadigan elementlar darhol yuklanishi kerak). Lekin "scroll event alternative" deb ishlatsangiz, cheksiz bu behavior kutilmagan bo'ladi.

**Yechim:** Design qaroringizni shunga moslang: agar initial state kerak bo'lsa — default behavior foydali (lazy loading). Agar faqat "state change" kerak bo'lsa — flag bilan birinchi chaqiriqni skip qilish kerak. Har ikkala case ham common — ehtiyojingizga qarab tanlang.

### Gotcha 5: `history.pushState` `popstate` event'ni trigger qilmaydi

`history.pushState` / `history.replaceState` URL'ni o'zgartiradi va history stack'ga yozadi, lekin **`popstate` event'ni trigger qilmaydi**. `popstate` faqat **user navigation** (back/forward button, `history.back()`, `history.go()`) paytida fire bo'ladi. SPA router'larda bu juda keng tarqalgan xatolik manbai.

```javascript
// ❌ Xato tasavvur — pushState popstate ishga tushiradi deb o'ylash:
window.addEventListener("popstate", (e) => {
  console.log("Navigation:", location.pathname);
  renderPage(location.pathname);
});

history.pushState({}, "", "/about");
// ⚠️ popstate event FIRE BO'LMAYDI!
// renderPage() chaqirilmaydi, sahifa eski holda qoladi

// ✅ Yechim 1: Manual dispatch — popstate'ni qo'lda fire qilish
function navigate(url) {
  history.pushState({}, "", url);
  // Manual event dispatch
  window.dispatchEvent(new PopStateEvent("popstate", { state: history.state }));
}

// ✅ Yechim 2: Custom "routechange" event ishlatish
function navigate(url) {
  history.pushState({}, "", url);
  window.dispatchEvent(new CustomEvent("routechange", { detail: { url } }));
}

window.addEventListener("routechange", (e) => {
  renderPage(e.detail.url);
});
window.addEventListener("popstate", () => {
  // Back/forward navigation
  renderPage(location.pathname);
});

// ✅ Yechim 3: Zamonaviy Navigation API (Chrome 102+)
// popstate'dan aniqroq API — barcha navigation turlarini ushlaydi
if ("navigation" in window) {
  navigation.addEventListener("navigate", (e) => {
    if (e.canIntercept && !e.destination.sameDocument) return;
    e.intercept({
      handler: async () => {
        await renderPage(e.destination.url);
      }
    });
  });
}
// Navigation API pushState, popstate, link click — hammasini bir joydan eshitadi
```

**Nima uchun:** HTML spec'i `pushState`'ni "URL o'zgartirish mexanizmi" sifatida belgilaydi, "navigation event" sifatida emas. `popstate` dizayn bo'yicha user navigation'ga reaksiya qilish uchun — shunda SPA'ning state'ini restore qilish kerak. `pushState` chaqirgan kod o'zi allaqachon yangi state'ni biladi, shuning uchun event kerak emas deb hisoblangan. Real hayotda bu taxmin noto'g'ri — route handler'lar ko'pincha pushState va popstate'ni bir xilda ishlash kerak bo'ladi.

**Yechim:** SPA router yozayotgan bo'lsangiz — manual dispatch pattern yoki custom event ishlating. Zamonaviy proyektda **Navigation API** (Chrome 102+, Firefox Nightly) ancha toza yechim — u pushState, popstate, link click'larni bir joyda ushlaydi. Safari'da hali yo'q — cross-browser uchun polyfill yoki fallback pattern kerak.

```javascript
function lazyLoad() {

  const observer = new IntersectionObserver((entries) => {
    entries.forEach(entry => {
      if (!entry.isIntersecting) return;
      const img = entry.target;

      img.src = img.dataset.src;
      img.removeAttribute("data-src");
```

```javascript
  function connect() {
    ws = new WebSocket(url);

    ws.addEventListener("open", () => {
      retries = 0;
      console.log("Chat ulandi");
    });

    ws.addEventListener("message", (e) => {
      onMessage(JSON.parse(e.data));
    });

    ws.addEventListener("close", () => {
      if (retries < 5) {
        setTimeout(connect, Math.min(1000 * 2 ** retries++, 30000));
      }
    });
  }

  connect();

  return {
    send(message) {
      if (ws.readyState === WebSocket.OPEN) {
        ws.send(JSON.stringify(message));
      }
    },
    close() {
      retries = 5;
      ws.close();
    }
  };
}
```

### Mashq 4: DOM Change Logger (Qiyin)

MutationObserver bilan barcha DOM o'zgarishlarini log qiling.

```javascript
function createDOMLogger(target = document.body) {
  const log = [];

  const observer = new MutationObserver((mutations) => {
    mutations.forEach(mutation => {
      const entry = {
        time: performance.now().toFixed(0),
        type: mutation.type,
        target: mutation.target.nodeName
      };

      if (mutation.type === "childList") {
        entry.added = [...mutation.addedNodes].filter(n => n.nodeType === 1).map(n => n.tagName);
        entry.removed = [...mutation.removedNodes].filter(n => n.nodeType === 1).map(n => n.tagName);
      } else if (mutation.type === "attributes") {
        entry.attribute = mutation.attributeName;
        entry.oldValue = mutation.oldValue;
      }

      log.push(entry);
      console.log("[DOM]", entry);
    });
  });

  observer.observe(target, {
    childList: true, attributes: true, characterData: true,
    subtree: true, attributeOldValue: true
  });

  return {
    getLog: () => [...log],
    stop: () => observer.disconnect()
  };
}
```
